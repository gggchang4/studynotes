> 适用范围：事务 ACID、事务状态、并发异常、调度与可串行化、冲突可串行化、封锁、隔离级别、两阶段锁协议、死锁处理、时间戳协议、乐观并发控制、MVCC、快照隔离、写偏序、多粒度锁、故障恢复、日志、检查点、ARIES。

**本章总口诀：**

```text
并发控制：先判断异常，再判断可串行化，再看协议是否能保证正确性。

```

## 1. 事务基础

### 1.1 事务定义

事务是访问并可能更新数据库的一个完整程序执行单元，具有 ACID 特性。

典型写法：

```sql
BEGIN TRANSACTION;
  read_item(A);
  A := A - 1000;
  write_item(A);
  read_item(B);
  B := B + 1000;
  write_item(B);
COMMIT;
```

答题句式：

```text
事务把一组数据库操作封装为一个逻辑工作单元，要么全部成功，要么全部撤销。
```

### 1.2 ACID

| 特性 | 含义 | 常考解释 |
| --- | --- | --- |
| 原子性 Atomicity | 事务内操作不可分割 | 要么全做，要么全不做 |
| 一致性 Consistency | 从一个一致状态到另一个一致状态 | 事务本身要保持完整性约束 |
| 隔离性 Isolation | 并发事务互不干扰 | 用户感觉不到其他事务并发执行 |
| 持久性 Durability | 提交后的修改永久保存 | 即使系统故障也不能丢失已提交结果 |

易错点：

```text
一致性由事务逻辑、完整性约束和并发控制共同保证；不是只靠 DBMS 自动保证。
```

### 1.3 事务状态

常见状态：

```text
活动状态 Active
  -> 部分提交状态 Partially Committed
  -> 提交状态 Committed

活动状态 Active
  -> 失败状态 Failed
  -> 中止状态 Aborted
```

解释模板：

```text
事务开始后进入活动状态；最后一条语句执行完但提交记录尚未稳定写入时是部分提交状态；
提交成功后进入提交状态；如果出错进入失败状态，随后执行回滚并进入中止状态。
```

## 2. 事务执行与正确性

### 2.1 调度

调度是多个事务中各个操作的执行顺序。

合法调度要求：

```text
1. 包含各事务的所有操作。
2. 每个事务内部操作顺序保持不变。
3. 不同事务之间的操作可以交错执行。
```

### 2.2 串行调度和非串行调度

串行调度：

```text
事务一个接一个执行，不交错。
```

非串行调度：

```text
多个事务操作交错执行。
```

判断句：

```text
非串行调度不一定错误；只要执行效果等价于某个串行调度，就是正确调度。
```

### 2.3 可串行化调度

定义：

```text
一个并发调度的结果与某个串行调度的结果相同，则称该调度是可串行化调度。
```

答题句式：

```text
可串行化是并发执行正确性的准则，它允许并发交错，但要求最终效果等价于某个串行执行顺序。
```

## 3. 并发异常识别模板

### 3.1 丢失更新 Lost Update

现象：

```text
两个事务都读取同一旧值，各自计算后写回，后写入者覆盖先写入者的结果。
```

判定模板：

```text
T1: read(A) ... write(A)
T2: read(A) ... write(A)
若 T2 读到的是 T1 写入前的旧值，且最终 T1 或 T2 的修改被覆盖，则是丢失更新。
```

答题句式：

```text
该调度中两个事务基于同一旧值修改同一数据项，后一次写入覆盖前一次写入，因此发生丢失更新。
```

### 3.2 脏读 Dirty Read

现象：

```text
T2 读取了 T1 尚未提交的写入结果，随后 T1 回滚。
```

判定模板：

```text
T1: write(A)
T2: read(A)
T1: rollback
```

结果：

```text
T2 读到的数据从未真正存在于数据库的提交状态中。
```

### 3.3 不可重复读 Non-repeatable Read

现象：

```text
同一事务两次读取同一数据项，中间被其他事务修改并提交，导致两次结果不同。
```

判定模板：

```text
T1: read(A) = old
T2: write(A), commit
T1: read(A) = new
```

### 3.4 幻象 Phantom

现象：

```text
同一事务两次执行同一范围查询，中间有其他事务插入/删除/修改满足条件的行，导致结果集合变化。
```

判定模板：

```text
T1: SELECT ... WHERE 条件
T2: INSERT / DELETE / UPDATE 使某行进入或离开该条件范围, commit
T1: 再次 SELECT 同一条件，行集合变化
```

### 3.5 异常识别速表

| 异常 | 看什么 | 关键词 |
| --- | --- | --- |
| 丢失更新 | 两个事务写同一项，后写覆盖先写 | 基于旧值、覆盖 |
| 脏读 | 读了别人未提交数据，别人后来回滚 | 未提交、rollback |
| 不可重复读 | 同一事务两次读同一行值不同 | 同一行、两次读 |
| 幻象 | 同一范围查询两次结果行数不同 | 范围查询、插入/删除 |

## 4. 冲突可串行化

### 4.1 冲突操作

两个操作冲突的条件：

```text
1. 来自不同事务；
2. 访问同一数据项；
3. 至少一个操作是 write。
```

冲突类型：

```text
read_i(X) 与 write_j(X)
write_i(X) 与 read_j(X)
write_i(X) 与 write_j(X)
```

不冲突：

```text
read_i(X) 与 read_j(X)
访问不同数据项的操作
同一事务内部操作
```

### 4.2 冲突等价

定义：

```text
如果两个调度中所有冲突操作的相对顺序都相同，则两个调度冲突等价。
```

答题句式：

```text
调度 S 与串行调度 S' 中每一对冲突操作的先后顺序一致，所以 S 与 S' 冲突等价。
```

### 4.3 优先图判定法

步骤：

```text
1. 每个事务画一个节点。
2. 从左到右扫描调度。
3. 若 Ti 的某操作 Oi 在 Tj 的冲突操作 Oj 之前，则加边 Ti -> Tj。
4. 扫描完后检查图中是否有环。
5. 无环：冲突可串行化；拓扑排序就是等价串行顺序。
6. 有环：不是冲突可串行化。
```

边的来源：

```text
ri(X) ... wj(X)  加 Ti -> Tj
wi(X) ... rj(X)  加 Ti -> Tj
wi(X) ... wj(X)  加 Ti -> Tj
```

答题模板：

```text
冲突对如下：
1. ...，因此加边 Ti -> Tj。
2. ...，因此加边 Tj -> Tk。

优先图中不存在环，所以该调度是冲突可串行化调度。
一个等价串行顺序为：...
```

如果有环：

```text
优先图中存在环 Ti -> Tj -> ... -> Ti，因此不存在满足这些先后约束的串行顺序，
该调度不是冲突可串行化调度。
```

### 4.4 典型易错点

- 只看不同事务之间的冲突。
- `read-read` 不冲突。
- 同一数据项才可能冲突。
- 加边方向是“先发生的冲突操作所在事务 -> 后发生的冲突操作所在事务”。
- 无环时可以有多个拓扑序，任取一个作为等价串行顺序。

## 5. 基础封锁与隔离级别

### 5.1 S 锁和 X 锁

| 锁 | 名称 | 含义 |
| --- | --- | --- |
| `SLock(A)` | 共享锁 | 读锁，允许多个事务同时读 |
| `XLock(A)` | 排他锁 | 写锁，独占数据项 |
| `Unlock(A)` | 解锁 | 释放数据项上的锁 |

兼容矩阵：

| 已有锁 \ 请求锁 | X | S | 无锁 |
| --- | --- | --- | --- |
| X | N | N | Y |
| S | N | Y | Y |
| 无锁 | Y | Y | Y |

使用原则：

```text
读前先申请 S 锁；写前先申请 X 锁；操作结束或事务结束后按协议释放锁。
```

### 5.2 一级、二级、三级封锁协议

一级封锁协议：

```text
事务在修改数据之前必须加 X 锁，并保持到事务结束。
解决：丢失更新。
```

二级封锁协议：

```text
一级封锁 + 事务读数据前加 S 锁，读完即可释放。
解决：脏读。
```

三级封锁协议：

```text
一级封锁 + 事务读数据前加 S 锁，并保持到事务结束。
解决：不可重复读。
```

速记：

```text
一级：写锁到结束。
二级：一级 + 读锁读完放。
三级：一级 + 读锁到结束。
```

### 5.3 SQL 隔离级别与异常

| 隔离级别 | 丢失更新 | 脏读 | 不可重复读 | 幻象 |
| --- | --- | --- | --- | --- |
| 读未提交 Read Uncommitted | 不防 | 可能 | 可能 | 可能 |
| 读已提交 Read Committed | 不防 | 不会 | 可能 | 可能 |
| 可重复读 Repeatable Read | 不防 | 不会 | 不会 | 可能 |
| 串行化 Serializable | 不会 | 不会 | 不会 | 不会 |

课件对应的封锁协议：

| 隔离级别 | 协议 |
| --- | --- |
| 读未提交 | 一级封锁协议 |
| 读已提交 | 二级封锁协议 |
| 可重复读 | 三级封锁协议 |

答题句式：

```text
提高隔离级别可以减少并发异常，但会降低并发度并增加等待、死锁或回滚的可能。
```

## 6. 两阶段锁协议 2PL

### 6.1 定义

两阶段锁协议要求事务的加锁和解锁分为两个阶段：

```text
增长阶段 Growing Phase：只能申请锁，不能释放锁。
收缩阶段 Shrinking Phase：只能释放锁，不能再申请新锁。
```

判断模板：

```text
找到事务第一次 Unlock 的位置。
如果该事务在第一次 Unlock 之后又 Lock 了任何数据项，则违反 2PL。
否则满足 2PL。
```

### 6.2 2PL 与冲突可串行化

定理：

```text
如果所有事务都遵守两阶段锁协议，则调度一定是冲突可串行化的。
```

但注意：

```text
2PL 是冲突可串行化的充分条件，不是必要条件。
有些可串行化调度不一定由 2PL 产生。
```

2PL 的串行化顺序：

```text
按事务的封锁点 lock point 排序。
封锁点：事务获得最后一个锁的时刻。
```

答题句式：

```text
由于每个事务遵守 2PL，其所有冲突依赖不会形成环，因此该调度冲突可串行化。
等价串行顺序可按各事务封锁点的先后确定。
```

### 6.3 2PL 的问题

可能出现：

- 死锁。
- 级联回滚。

死锁例子：

```text
T1 持有 A，等待 B。
T2 持有 B，等待 A。
两者互相等待，形成死锁。
```

级联回滚例子：

```text
T1 写 A 后释放 X 锁，T2 读 A。
如果 T1 后来回滚，则 T2 读到了无效数据，T2 也必须回滚。
```

### 6.4 严格 2PL 与强严格 2PL

严格 2PL：

```text
在遵守 2PL 的基础上，事务持有的 X 锁必须到事务提交或回滚后才释放。
```

作用：

```text
避免其他事务读到未提交写入，从而减少级联回滚。
```

强严格 2PL：

```text
在遵守 2PL 的基础上，事务持有的所有锁都必须到事务提交或回滚后才释放。
```

比较：

| 协议 | 保持到结束的锁 | 作用 |
| --- | --- | --- |
| 2PL | 不要求所有锁到结束 | 保证冲突可串行化，可能死锁/级联回滚 |
| 严格 2PL | X 锁 | 防止读未提交写入，减少级联回滚 |
| 强严格 2PL | S 锁和 X 锁 | 更强隔离，恢复更简单，并发度更低 |

## 7. 死锁处理模板

### 7.1 死锁检测

事务等待图：

```text
节点：事务。
边 Ti -> Tj：Ti 正在等待 Tj 持有的锁。
```

判定：

```text
等待图有环 => 存在死锁。
等待图无环 => 当前无死锁。
```

处理：

```text
选择一个或多个事务作为牺牲者，回滚并释放其锁。
```

### 7.2 死锁预防策略

No-Wait：

```text
如果请求的锁已被其他事务持有，则请求事务不等待，直接回滚。
```

Wait-Die：

```text
老事务请求年轻事务持有的锁：老事务等待。
年轻事务请求老事务持有的锁：年轻事务回滚。
```

速记：

```text
老等少，少者死。
```

Wound-Wait：

```text
老事务请求年轻事务持有的锁：年轻事务被抢占并回滚。
年轻事务请求老事务持有的锁：年轻事务等待。
```

速记：

```text
老抢少，少等老。
```

优先级：

```text
通常按事务开始时间决定优先级：开始越早，优先级越高。
事务重启时保持原时间戳，可减少饥饿。
```

### 7.3 死锁题答题模板

```text
1. 列出每个事务当前持有的锁。
2. 列出每个事务正在请求但未获得的锁。
3. 画等待边：请求者 -> 持有者。
4. 若等待图有环，说明死锁。
5. 根据题目指定策略：
   no-wait：请求失败者回滚。
   wait-die：年轻请求老则回滚，老请求年轻则等待。
   wound-wait：老请求年轻则年轻回滚，年轻请求老则等待。
```

## 8. 时间戳协议 TO

### 8.1 基本思想

时间戳协议给每个事务分配单调递增的时间戳：

```text
TS(T)：事务 T 的开始时间戳。
若 TS(Ti) < TS(Tj)，则 Ti 应该排在 Tj 之前。
```

TO 的目标：

```text
所有冲突操作的执行顺序必须与事务时间戳顺序一致。
```

### 8.2 数据项时间戳

每个数据项 `x` 维护：

```text
RTS(x)：最近一次成功读取 x 的事务时间戳。
WTS(x)：最近一次成功写入 x 的事务时间戳。
```

### 8.3 读操作规则

事务 `Ti` 读 `x`：

```text
如果 TS(Ti) < WTS(x)：
  说明 Ti 试图读取一个“未来事务”已写过的数据，违反时间戳顺序，Ti 回滚。
否则：
  允许读 x；
  RTS(x) = max(RTS(x), TS(Ti))。
```

记忆：

```text
读看 WTS。
```

### 8.4 写操作规则

事务 `Ti` 写 `x`：

```text
如果 TS(Ti) < RTS(x)：
  说明已经有更年轻事务读过 x，Ti 再写会破坏顺序，Ti 回滚。
否则如果 TS(Ti) < WTS(x)：
  说明已经有更年轻事务写过 x，Ti 再写会破坏顺序，Ti 回滚。
否则：
  允许写 x；
  WTS(x) = max(WTS(x), TS(Ti))。
```

记忆：

```text
写看 RTS 和 WTS。
```

### 8.5 Thomas 写规则

Thomas 写规则是 TO 的特殊优化：

```text
若 TS(Ti) < WTS(x)，则 Ti 对 x 的写入已经过时，可以忽略该写操作，而不是回滚整个事务。
```

但仍要注意：

```text
若 TS(Ti) < RTS(x)，通常仍需回滚，因为已有更年轻事务读过旧值。
```

### 8.6 TO 判定题模板

```text
1. 写出每个事务的 TS。
2. 写出每个数据项的 RTS、WTS 初值。
3. 从调度开头逐个处理操作。
4. 遇到读 Ri(x)：比较 TS(Ti) 与 WTS(x)。
5. 遇到写 Wi(x)：比较 TS(Ti) 与 RTS(x)、WTS(x)。
6. 若检查不通过，说明 Ti 回滚；否则执行并更新 RTS/WTS。
```

答题句式：

```text
在操作 Wi(x) 处，TS(Ti) 小于 RTS(x)，说明已有更晚事务读取过 x。
若允许 Ti 写入会破坏时间戳顺序，因此 Ti 必须回滚。
```

### 8.7 TO 的级联回滚问题

TO 可能产生级联回滚：

```text
T1 写 x 后尚未提交，T2 读取 x。
若 T1 后续回滚，则 T2 也必须回滚。
```

解决思路：

```text
维护恢复列表，或采用预写操作，让写入先进入私有空间，提交时再真正写入数据库。
```

## 9. 乐观并发控制 OCC

### 9.1 基本思想

OCC 认为冲突较少，因此先不加锁执行，提交前再检查冲突。

三个阶段：

```text
1. 读取阶段：读数据库，写入事务私有空间。
2. 验证阶段：检查该事务是否与其他事务冲突。
3. 写阶段：若验证通过，将私有空间中的更新写入数据库。
```

注意：

```text
同一时刻通常只允许一个事务进入验证和写阶段，以便简化验证。
```

### 9.2 OCC 的数据结构

每个事务 `Ti` 维护：

```text
Ti.RS：读集，Ti 读取过的数据项集合。
Ti.WS：写集，Ti 写过的数据项集合。
S-TS(Ti)：开始时间戳。
V-TS(Ti)：进入验证阶段的时间戳。
F-TS(Ti)：完成/提交时间戳。
```

### 9.3 OCC 验证规则

课件中的核心检查：

```text
检查对象：
  Ti 开始时尚未提交、Ti 进入验证时已经完成提交的事务 Tj。

检查条件：
  Ti.RS ∩ Tj.WS = ∅
```

含义：

```text
如果 Ti 读过的数据被并发事务 Tj 写过，则 Ti 读到的可能是过期值，应回滚 Ti。
```

### 9.4 OCC 判定题模板

```text
1. 列出当前验证事务 Ti 的读集 Ti.RS 和写集 Ti.WS。
2. 找出需要检查的事务 Tj：与 Ti 时间重叠，且在 Ti 验证前已经提交。
3. 对每个 Tj 计算 Ti.RS ∩ Tj.WS。
4. 若交集为空，Ti 通过验证。
5. 若交集非空，Ti 回滚。
```

答题句式：

```text
由于 Ti.RS 与 Tj.WS 存在交集，说明 Ti 读取的数据已被 Tj 并发修改，
因此 Ti 的读结果不再可靠，验证失败并回滚。
```

### 9.5 OCC 总结

优点：

- 读阶段不加锁，读写并发度高。
- 无死锁。
- 适合读多写少、冲突少的场景。

缺点：

- 冲突多时提交阶段回滚频繁。
- 长事务容易验证失败。
- 写阶段可能成为瓶颈。

## 10. 多版本并发控制 MVCC

### 10.1 基本思想

MVCC 保存同一逻辑对象的多个物理版本。

规则：

```text
写事务：创建一个新版本。
读事务：根据时间戳或快照选择可见版本。
```

主要效果：

- 读不阻塞写。
- 写不阻塞读。
- 只读事务可读取一致快照。
- 支持 time-travel 查询。

### 10.2 MVCC 实现要点

需要考虑：

| 模块 | 要解决的问题 |
| --- | --- |
| 并发控制协议 | 哪个版本对哪个事务可见 |
| 版本存储 | 多版本如何存储 |
| 垃圾回收 | 旧版本何时可以删除 |
| 索引管理 | 索引指向版本链头还是具体版本 |

版本存储方案：

```text
1. 追加存储 Append-Only Storage：每次更新追加新版本。
2. 时间旅行存储 Time-Travel Storage：单独维护历史版本。
3. 增量存储 Delta Storage：保存当前版本和版本间差异。
```

垃圾回收：

```text
对象级垃圾回收：清理某个对象上不可见的旧版本。
事务级垃圾回收：事务结束后清理该事务产生且不再可见的版本。
```

索引管理：

```text
逻辑指针：索引指向逻辑对象，再由版本链找到可见版本。
物理指针：索引直接指向具体版本，查询更直接但维护成本更高。
```

### 10.3 快照隔离 SI

快照隔离：

```text
事务开始时获得数据库的一致快照，之后读操作都基于该快照。
写操作先写入私有空间，提交时再验证是否存在冲突。
```

三个阶段：

```text
1. 执行阶段：读取快照，写入私有工作区。
2. 验证阶段：提交前检查写冲突。
3. 提交阶段：验证通过则写入数据库；否则回滚。
```

只读事务：

```text
通常可以忽略验证阶段，因为只读事务不修改数据库。
```

### 10.4 MVCC 与 SI 的一致性保证

课件表格可记为：

| 机制 | 不丢失修改 | 不读脏数据 | 可重复读 | 不幻读 | 不写偏斜 |
| --- | --- | --- | --- | --- | --- |
| MVCC | 不一定 | 可以 | 可以 | 可以 | 不一定 |
| SI | 可以 | 可以 | 可以 | 可以 | 不一定 |

重点：

```text
快照隔离通常可以避免脏读、不可重复读和幻读，也能避免典型丢失更新；
但它不等价于真正的可串行化，因为可能发生写偏斜。
```

### 10.5 写偏序 / 写偏斜 Write Skew

典型结构：

```text
约束：A 和 B 至少一个为真。
T1 读 A、B，看到 B 为真，于是把 A 改为假。
T2 读 A、B，看到 A 为真，于是把 B 改为假。
两个事务写不同对象，因此 SI 可能都允许提交。
最终 A、B 都为假，违反约束。
```

答题句式：

```text
两个事务读取了相同快照，并基于该快照分别更新不同数据项。
由于写集不冲突，SI 允许二者提交，但全局约束被破坏，因此发生写偏斜。
```

## 11. 多粒度锁

### 11.1 为什么需要多粒度锁

只锁大对象：

```text
锁少，管理开销低，但并发度低。
```

只锁小对象：

```text
并发度高，但需要大量锁，管理开销高。
```

多粒度锁：

```text
把数据库对象组织成层次结构，例如 数据库 -> 关系 -> 块 -> 元组。
根据操作范围选择合适粒度加锁。
```

### 11.2 意向锁

| 锁 | 含义 |
| --- | --- |
| `IS` | 意向共享锁，表示将在下层对象上加 S 锁 |
| `IX` | 意向排他锁，表示将在下层对象上加 X 锁 |
| `SIX` | 当前节点 S 锁 + 下层意向 X 锁 |

加锁规则：

```text
1. 从层次结构根节点开始向下加锁。
2. 若要在某节点加 S 锁，必须先在祖先节点加 IS 锁。
3. 若要在某节点加 X 锁，必须先在祖先节点加 IX 锁。
4. 若已在某节点加锁，则继续向目标子节点前进。
5. 解锁通常从叶子向根释放。
```

### 11.3 意向锁兼容矩阵

常用矩阵：

| 已有 \ 请求 | IS | IX | S | SIX | X |
| --- | --- | --- | --- | --- | --- |
| IS | Y | Y | Y | Y | N |
| IX | Y | Y | N | N | N |
| S | Y | N | Y | N | N |
| SIX | Y | N | N | N | N |
| X | N | N | N | N | N |

记忆：

```text
IS 很温和，除了 X 基本都兼容；
IX 表示下层可能写，所以与 S/SIX/X 不兼容；
X 最强，和任何锁都不兼容。
```

### 11.4 多粒度锁题模板

```text
1. 画出层次结构：关系 -> 块 -> 元组。
2. 判断事务最终要读还是写哪个层次的对象。
3. 从根节点开始加意向锁：
   读下层：祖先加 IS。
   写下层：祖先加 IX。
4. 到目标节点后加 S 或 X。
5. 检查每一层的锁兼容性。
6. 若某层不兼容，则事务等待或按死锁策略处理。
```

## 12. 数据库故障与恢复

### 12.1 为什么需要恢复

恢复子系统的目标：

```text
无论发生哪种故障，都能把数据库恢复到某个已知正确状态。
```

没有恢复机制的后果：

```text
数据可能出错或丢失，破坏原子性和持久性。
```

### 12.2 故障分类

| 故障 | 含义 | 常见处理 |
| --- | --- | --- |
| 事务故障 | 单个事务未到提交/回滚终点就终止 | Undo 该事务 |
| 系统故障 | 系统崩溃、断电、内存丢失，但磁盘未坏 | Undo 未提交事务，Redo 已提交事务 |
| 介质故障 | 磁盘损坏、磁头碰撞等 | 用备份恢复，再用日志追赶 |
| 用户错误 | 误删表、误删数据、提交错误数据 | 闪回、备份、日志恢复 |

### 12.3 事务故障

事务故障含义：

```text
事务没有达到预期终点 COMMIT 或显式 ROLLBACK 就终止。
```

恢复：

```text
执行事务撤销 UNDO，撤销该事务已经对数据库做出的修改，
使数据库像该事务从未启动一样。
```

### 12.4 系统故障

系统故障影响：

- 正在运行的事务全部异常终止。
- 内存和缓冲区内容丢失。
- 磁盘数据库本身未损坏。

恢复策略：

```text
1. 撤销所有未完成事务。
2. 重做所有已提交事务。
```

### 12.5 介质故障

介质故障：

```text
外存设备损坏，导致数据库全部或部分数据丢失。
```

恢复策略：

```text
1. 装入最近一次备份。
2. 根据日志重做备份之后已经提交的事务。
3. 必要时撤销未完成事务。
```

## 13. 日志与 WAL

### 13.1 日志文件内容

日志记录通常包含：

```text
Transaction Id：事务 ID
Object Id：对象 ID
Before Value：旧值，用于 UNDO
After Value：新值，用于 REDO
```

事务日志还记录：

```text
<BEGIN T>
<COMMIT T>
<ROLLBACK T>
<T, X, old, new>
```

### 13.2 物理日志与逻辑日志

物理日志：

```text
记录某个数据项的旧值和新值。
例如：A 从 100 改为 80。
```

逻辑日志：

```text
记录更高层操作。
例如：执行 UPDATE、DELETE、INSERT 语句。
```

比较：

```text
物理日志恢复直接，日志量可能较大；
逻辑日志日志量可能较小，但恢复时需要重新解释操作。
```

### 13.3 Undo 日志

格式：

```text
<T, X, v>
```

含义：

```text
事务 T 把 X 从旧值 v 改走，日志保存旧值 v，用于撤销。
```

基本规则：

```text
1. <T, X, v> 必须先写入磁盘，然后 X 的新值才能写入磁盘。
2. 若事务提交，COMMIT 日志必须在该事务更新的数据元素写盘之后写入磁盘。
```

恢复：

```text
对未提交事务，根据日志中的旧值执行 UNDO。
```

### 13.4 Redo 日志

格式：

```text
<T, X, w>
```

含义：

```text
事务 T 为 X 写入新值 w，日志保存新值 w，用于重做。
```

恢复：

```text
对已提交但数据页可能尚未写盘的事务，根据日志中的新值执行 REDO。
```

### 13.5 Undo/Redo 日志

格式：

```text
<T, X, v, w>
```

含义：

```text
旧值 v 用于 UNDO，新值 w 用于 REDO。
```

适用：

```text
系统既允许未提交事务的数据页落盘，又不强制提交时立即把所有数据页落盘。
```

### 13.6 WAL 原则

WAL：

```text
Write-Ahead Logging，预写日志。
```

核心规则：

```text
1. 数据页写入磁盘前，描述该修改的日志记录必须先写入磁盘。
2. 事务提交前，该事务相关日志必须先写入磁盘。
```

为什么要先写日志：

```text
如果数据页改了但没有日志，故障后无法撤销或重做；
如果日志先写了但数据页没写，故障后仍可根据日志恢复。
```

日志刷盘时机：

- 日志缓冲区达到阈值。
- 脏数据页准备写入磁盘。
- 事务提交。

## 14. 备份与基本恢复

### 14.1 备份分类

按数据组织维度：

```text
逻辑备份：导出表、模式、记录等逻辑对象。
物理备份：复制数据文件、日志文件、控制文件等物理文件。
```

按备份时机：

```text
静态备份：备份期间数据库不运行事务。
动态备份：备份期间事务仍可运行，需要结合日志。
```

按备份规模：

```text
全量备份：备份全部数据。
增量备份：只备份上次备份后变化的数据。
```

### 14.2 静态备份与动态备份

静态备份：

```text
备份开始时数据库处于一致状态，备份期间不允许修改。
优点：实现简单。
缺点：降低可用性。
```

动态备份：

```text
备份与事务并发执行，备份数据可能不是单独一致的快照。
必须结合日志，才能恢复到某一时刻的一致状态。
```

### 14.3 事务故障恢复模板

```text
1. 反向扫描日志，找到该事务的更新记录。
2. 对每个更新记录执行逆操作。
3. 继续反向扫描该事务其他更新记录。
4. 直到找到该事务的 BEGIN 记录。
5. 写入回滚完成记录。
```

一句话：

```text
事务故障只需要撤销该事务本身已经做过的更新。
```

### 14.4 系统故障恢复模板

```text
1. 正向扫描日志，找出已提交事务和未完成事务。
2. 对已提交但可能未落盘的事务执行 REDO。
3. 对故障时仍未提交的事务执行 UNDO。
4. 恢复数据库到故障前的正确状态。
```

### 14.5 检查点恢复

检查点记录内容：

```text
1. 建立检查点时仍在运行的事务清单。
2. 这些事务最近一条日志记录的位置。
```

检查点作用：

```text
减少故障恢复时需要扫描的日志范围。
```

创建检查点步骤：

```text
1. 将当前日志缓冲区中的日志记录写入磁盘。
2. 写入检查点记录。
3. 将当前脏数据页按策略写盘。
4. 更新重启文件或 MasterRecord，使其指向最新检查点。
```

根据检查点判断：

```text
若事务在检查点之前提交，通常无需重做。
若事务在检查点之后提交，可能需要重做。
若事务在故障时仍未提交，需要撤销。
```

## 15. 事务缓冲区策略

### 15.1 STEAL / NO-STEAL

从未提交事务角度：

```text
STEAL：允许未提交事务修改过的脏页写入磁盘。
NO-STEAL：不允许未提交事务修改过的脏页写入磁盘。
```

影响：

```text
STEAL 可能让未提交修改进入磁盘，因此故障恢复需要 UNDO。
NO-STEAL 不会把未提交修改写盘，因此通常不需要 UNDO。
```

### 15.2 FORCE / NO-FORCE

从已提交事务角度：

```text
FORCE：事务提交时强制把该事务修改过的数据页写入磁盘。
NO-FORCE：事务提交时不强制把该事务修改过的数据页写入磁盘。
```

影响：

```text
FORCE 提交后数据已落盘，通常不需要 REDO。
NO-FORCE 提交后数据可能只在缓冲区，故障恢复需要 REDO。
```

### 15.3 策略组合表

| 策略 | 正常运行性能 | 恢复需要 |
| --- | --- | --- |
| NO-STEAL + FORCE | 较差 | 通常不需要 UNDO/REDO |
| STEAL + FORCE | 较好 | 需要 UNDO |
| NO-STEAL + NO-FORCE | 较好 | 需要 REDO |
| STEAL + NO-FORCE | 最常用，正常性能好 | 需要 UNDO + REDO |

课件重点：

```text
商业数据库通常采用 STEAL + NO-FORCE，因此恢复算法需要同时支持 UNDO 和 REDO。
```

## 16. ARIES 算法

### 16.1 ARIES 基本特点

ARIES 适合：

```text
STEAL + NO-FORCE 缓冲策略。
```

核心数据结构：

| 名称 | 含义 |
| --- | --- |
| `LSN` | 日志序列号，每条日志单调递增 |
| `pageLSN` | 数据页上最近一次修改对应的 LSN |
| `flushedLSN` | 已经刷到磁盘的最大日志 LSN |
| `prevLSN` | 同一事务上一条日志记录 |
| `UndoNextLSN` | CLR 中指向下一条需要撤销的日志 |
| `ATT` | Active Transaction Table，活跃事务表 |
| `DPT` | Dirty Page Table，脏页表 |
| `recLSN` | 某页第一次变脏对应的日志 LSN |

WAL 在 ARIES 中的体现：

```text
数据页写盘前必须保证 pageLSN <= flushedLSN。
```

### 16.2 CLR 补偿日志

CLR：

```text
Compensation Log Record，补偿日志记录。
```

作用：

```text
记录事务回滚过程中已经执行的撤销操作。
CLR 只需要 REDO，不需要再 UNDO。
```

好处：

```text
如果系统在回滚过程中再次崩溃，恢复时可以知道哪些撤销已经完成，避免重复撤销。
```

### 16.3 模糊检查点

普通检查点可能需要等待所有事务提交并阻塞新事务，影响性能。

模糊检查点：

```text
创建检查点时不阻塞正在执行的事务，只记录当时的 ATT 和 DPT 等状态。
```

基本步骤：

```text
1. 写入 <CHECKPOINT_BEGIN> 日志。
2. 将可安全写出的数据页刷盘。
3. 正在修改的数据页继续保留在 DPT 中。
4. 未提交事务加入 ATT。
5. 检查点结束后写入 <CHECKPOINT_END> 日志。
6. MasterRecord 指向最新成功检查点。
```

### 16.4 ARIES 恢复三阶段

Analysis Phase：

```text
从 MasterRecord 指向的检查点开始扫描日志到末尾，
重建 ATT 和 DPT，
确定哪些事务是 loser transaction，哪些脏页可能需要重做。
```

Redo Phase：

```text
从 DPT 中最小的 recLSN 开始向前扫描日志，
重复历史，重做需要重做的操作。
```

Undo Phase：

```text
根据 ATT，反向撤销故障时未完成的事务，
每撤销一步写一条 CLR。
```

### 16.5 Redo 是否需要执行的判断

对某条更新日志 `LSN`，通常满足以下条件才需要实际重做：

```text
1. 对应页面在 DPT 中；
2. LSN >= 该页面的 recLSN；
3. 页面磁盘版本的 pageLSN < 当前日志 LSN。
```

如果：

```text
pageLSN >= LSN
```

说明该更新已经在磁盘页上，不必重复执行。

### 16.6 ARIES 题模板

```text
1. 从检查点开始做 Analysis：
   - 遇到 BEGIN/UPDATE，更新 ATT。
   - 遇到 COMMIT/END，更新事务状态。
   - 遇到更新页面，若页面不在 DPT，则加入 DPT 并设置 recLSN。

2. Redo：
   - 从 DPT 中最小 recLSN 对应日志开始。
   - 对每条更新日志判断是否需要重做。
   - 若需要，执行 REDO 并更新页面。

3. Undo：
   - 找出 ATT 中未完成事务。
   - 按 LSN 从大到小回滚。
   - 每撤销一条普通更新日志，写 CLR。
   - 根据 prevLSN 或 UndoNextLSN 继续。
   - 事务撤销完写 END。
```

## 17. 高频题型模板

### 17.1 判断并发异常

```text
1. 找同一数据项上的读写顺序。
2. 判断是否存在：
   - 两个写覆盖：丢失更新。
   - 读未提交后对方回滚：脏读。
   - 同一事务两次读同一项不同：不可重复读。
   - 同一范围两次查询集合不同：幻象。
3. 写出异常名称和原因。
```

### 17.2 判断调度是否冲突可串行化

```text
1. 列事务节点。
2. 扫描所有冲突操作。
3. 对每个冲突对加边。
4. 检查优先图是否有环。
5. 无环：可串行化，给出拓扑序。
6. 有环：不可冲突串行化。
```

### 17.3 判断是否满足 2PL

```text
1. 对每个事务单独检查。
2. 标出该事务第一次 Unlock。
3. 如果第一次 Unlock 后还有 Lock，则违反 2PL。
4. 如果所有事务都不违反，则满足 2PL。
5. 满足 2PL 可推出冲突可串行化。
```

### 17.4 判断隔离级别

```text
1. 看是否允许读未提交：允许则读未提交。
2. 看是否避免脏读但两次读可能不同：读已提交。
3. 看是否同一行可重复读但范围可能出现幻象：可重复读。
4. 看是否与串行执行等价、范围查询也稳定：串行化。
```

### 17.5 时间戳协议判定

```text
1. 写出 TS(Ti)。
2. 写出每个数据项 RTS/WTS。
3. 读操作：若 TS(Ti) < WTS(x)，Ti 回滚；否则更新 RTS。
4. 写操作：若 TS(Ti) < RTS(x) 或 TS(Ti) < WTS(x)，Ti 回滚；否则更新 WTS。
5. 若使用 Thomas 写规则，过时写可忽略。
```

### 17.6 OCC 验证

```text
1. 列出 Ti.RS、Ti.WS。
2. 找出与 Ti 重叠且已提交的 Tj。
3. 检查 Ti.RS ∩ Tj.WS。
4. 交集为空：Ti 提交。
5. 交集非空：Ti 回滚。
```

### 17.7 MVCC / SI 判定

```text
1. 读操作读取事务开始时可见的版本。
2. 写操作生成新版本。
3. 提交时检查是否存在写写冲突。
4. 若写同一对象冲突，后提交者回滚。
5. 若两个事务写不同对象但共同破坏约束，说明可能发生写偏斜。
```

### 17.8 日志恢复题

```text
1. 区分事务状态：
   已提交事务：需要 REDO。
   未提交事务：需要 UNDO。
2. 区分日志类型：
   Undo 日志用旧值撤销。
   Redo 日志用新值重做。
   Undo/Redo 日志二者都可做。
3. 有检查点时，从检查点附近开始，不必从日志开头全扫。
4. 系统故障恢复：Redo committed，Undo uncommitted。
```

### 17.9 ARIES 恢复题

```text
1. Analysis：重建 ATT 和 DPT。
2. Redo：从最小 recLSN 开始，重复历史。
3. Undo：回滚 ATT 中未完成事务，写 CLR。
4. 判断是否 redo 某页：看 DPT、recLSN、pageLSN。
5. 判断是否可刷页：pageLSN <= flushedLSN。
```

## 18. 易错清单

- 非串行调度不一定错误，可串行化才是正确性准则。
- 冲突必须满足“不同事务、同一数据项、至少一个写”。
- 优先图边的方向是先发生冲突操作的事务指向后发生的事务。
- 2PL 能保证冲突可串行化，但不能避免所有死锁。
- 严格 2PL 重点是 X 锁到提交；强严格 2PL 是所有锁到提交。
- `S` 锁之间兼容，`X` 锁与任何 `S/X` 都不兼容。
- 读已提交只能防脏读，不能防不可重复读。
- 可重复读通常还不能完全防幻象。
- Wait-Die 是老事务等待、年轻事务回滚。
- Wound-Wait 是老事务抢占年轻事务、年轻事务等待。
- TO 读操作只看 `WTS(x)`，写操作看 `RTS(x)` 和 `WTS(x)`。
- OCC 不是不检查冲突，而是提交前再检查。
- MVCC 不等于可串行化；SI 可能发生写偏斜。
- 多粒度锁必须先在祖先节点加意向锁，再在目标节点加普通锁。
- STEAL 意味着可能需要 UNDO；NO-FORCE 意味着可能需要 REDO。
- WAL 的核心是日志先于数据页落盘。
- ARIES 的 CLR 只 redo，不再 undo。
- ARIES Redo 是“重复历史”，Undo 才真正回滚 loser 事务。

## 19. 一页速记版

```text
ACID：
原子性 全做或全不做
一致性 一致状态到一致状态
隔离性 并发互不干扰
持久性 提交永久保存

异常：
丢失更新：两个写覆盖
脏读：读未提交后对方回滚
不可重复读：同一行两次读不同
幻象：同一范围两次结果集不同

冲突可串行化：
不同事务 + 同一数据项 + 至少一个写
建优先图，无环则可串行化

封锁：
S-S 兼容；X 与 S/X 不兼容
一级：X 锁写到结束
二级：一级 + S 锁读完放
三级：一级 + S 锁到结束

2PL：
先加锁后解锁，第一次 Unlock 后不能再 Lock
严格 2PL：X 锁到提交
强严格 2PL：所有锁到提交

死锁：
等待图有环
No-Wait：不等，直接回滚
Wait-Die：老等少，少者死
Wound-Wait：老抢少，少等老

TO：
读 Ri(x)：TS(Ti) < WTS(x) 则回滚，否则 RTS=max(RTS,TS)
写 Wi(x)：TS(Ti) < RTS(x) 或 TS(Ti) < WTS(x) 则回滚，否则 WTS=max(WTS,TS)

OCC：
读阶段 -> 验证阶段 -> 写阶段
验证：Ti.RS ∩ Tj.WS = 空

MVCC/SI：
读旧版本，写新版本
SI 防脏读/不可重复读/幻象/丢失更新，但可能写偏斜

多粒度锁：
IS 下层读，IX 下层写，SIX 当前读且下层写
自根向下加锁，自叶向上解锁

恢复：
事务故障：UNDO
系统故障：UNDO 未提交 + REDO 已提交
介质故障：备份 + 日志
STEAL 需要 UNDO；NO-FORCE 需要 REDO
WAL：日志先于数据页落盘

ARIES：
Analysis 重建 ATT/DPT
Redo 从最小 recLSN 重复历史
Undo 回滚 loser，写 CLR
pageLSN <= flushedLSN 才能刷页
```
