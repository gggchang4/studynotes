# 第08章 索引：开卷考试解题思路与模板

> 适用范围：索引概述、稠密/稀疏索引、多级索引、主索引/辅助索引、B+ 树、B-link 树、散列索引、可扩展散列、线性散列、位图索引、倒排索引、高维索引、LSM Tree。

## 0. 考场快速判题地图

| 题目关键词 | 题型 | 先套什么模板 |
| --- | --- | --- |
| 索引是什么、为什么快 | 概念题 | 用“键值 -> 指针/记录位置”解释 |
| 稠密索引、稀疏索引 | 比较题 | 每条记录一个索引项 vs 每块一个索引项 |
| 主索引、辅助索引 | 比较题 | 文件排序字段上的索引 vs 非排序字段上的索引 |
| 多级索引 | 计算/概念题 | 第一层索引太大，再对索引建索引 |
| B+ 树查找 | 过程题 | 根 -> 中间 -> 叶，叶节点链表支持范围查询 |
| B+ 树插入 | 操作题 | 叶插入，溢出则分裂并向父节点提升分隔键 |
| B+ 树删除 | 操作题 | 删除后不足则借用或合并，必要时调整父节点 |
| 散列索引 | 概念/操作题 | `h(key)` 定位桶，溢出用桶链 |
| 可扩展散列 | 操作题 | 目录位数、桶局部深度、桶分裂 |
| 线性散列 | 操作题 | 分裂指针按顺序扩展桶 |
| 位图索引 | 计算题 | OR、AND 位运算处理多条件查询 |
| 倒排索引 | 设计/查询题 | 词项 -> 文档列表，AND 取交集 |
| 高维索引 | 比较题 | 网格、分段散列、kd 树、四叉树、R 树 |
| LSM Tree | 比较题 | 更新密集，批量顺序写，后台合并 |

**索引题总口令：**

1. 先看查询条件：等值、范围、组合、多维、文本、更新密集。
2. 再选索引：B+ 树适合范围，哈希适合等值，位图适合低基数多条件，倒排适合文本，LSM 适合写密集。
3. 计算 I/O 时，先数索引层数，再数数据块访问。
4. 更新索引时，先定位叶/桶/列表，再判断是否溢出、合并或分裂。

## 1. 索引概述

### 1.1 索引定义

索引是对数据文件中记录的查找辅助结构。它通常保存：

```text
搜索码值 -> 指向记录或数据块的指针
```

查找过程：

```text
查询值 -> 查索引 -> 得到指针/块号 -> 读取满足条件的记录
```

没有索引时：

```text
可能需要扫描大量记录或全表扫描。
```

有索引时：

```text
先在较小的索引结构中定位，再访问少量数据块。
```

### 1.2 课件中的索引分类

| 分类 | 含义 |
| --- | --- |
| 稠密索引 | 每个搜索码值或记录都有索引项 |
| 稀疏索引 | 只对部分搜索码值建立索引项，通常每个数据块一个 |
| 多级索引 | 对索引再建索引 |
| 主索引 | 数据文件按搜索码有序，该排序字段上的索引 |
| 辅助索引 | 非文件排序字段上的索引 |
| 倒排索引 | 文本检索中，词项指向文档列表 |

### 1.3 索引优缺点

优点：

- 减少查询 I/O。
- 加快等值查询、范围查询、连接查询和排序。
- 支持唯一性约束和快速定位。

缺点：

- 占用额外存储空间。
- 插入、删除、更新时需要维护索引。
- 索引选择不当可能无明显收益。

答题句：

```text
索引本质上是用额外空间和维护代价换取查询效率。
```

## 2. 稠密索引、稀疏索引和多级索引

### 2.1 稠密索引

定义：

```text
对数据文件中的每个搜索码值都有一个索引项。
```

特点：

- 查找任意记录都能直接定位。
- 支持键值不存在时的精确判断。
- 索引项多，索引文件较大。
- 插入和删除时维护代价较高。

适用：

- 辅助索引通常是稠密索引。
- 对非排序字段查询。

### 2.2 稀疏索引

定义：

```text
只对部分搜索码值建立索引项，通常每个数据块或每个区间一个索引项。
```

查找过程：

```text
1. 在索引中找到小于等于查询值的最大索引项。
2. 根据指针找到对应数据块。
3. 在数据块中顺序查找目标记录。
```

特点：

- 索引项少，空间小。
- 维护代价较低。
- 查询需要在数据块内扫描。
- 要求数据文件按搜索码有序。

### 2.3 稠密 vs 稀疏

| 比较项 | 稠密索引 | 稀疏索引 |
| --- | --- | --- |
| 索引项数量 | 多 | 少 |
| 空间开销 | 大 | 小 |
| 查询定位 | 更直接 | 先定位块再块内查找 |
| 更新代价 | 较大 | 较小 |
| 是否要求数据有序 | 不一定 | 通常要求有序 |

### 2.4 多级索引

当一级索引太大，无法一次放入内存时，可对索引文件再建立索引：

```text
数据文件 -> 第一级索引 -> 第二级索引 -> ... -> 顶层索引
```

查找：

```text
从最高层索引开始逐层定位，直到数据文件。
```

I/O 估算模板：

```text
查询 I/O ≈ 索引层数 + 数据块访问数
```

课件口令：

```text
多级索引可以降低搜索代价，但插入、删除时维护更复杂。
```

## 3. 主索引、辅助索引与索引过滤

### 3.1 主索引

定义：

```text
数据文件按某搜索码有序，该搜索码上的索引称为主索引。
```

特点：

- 数据文件顺序与索引键顺序一致。
- 通常可以使用稀疏索引。
- 范围查询效率高。

### 3.2 辅助索引

定义：

```text
建立在非文件排序字段上的索引称为辅助索引。
```

特点：

- 数据文件没有按该字段排序。
- 通常需要稠密索引。
- 若同一键对应多条记录，可使用桶或中间接结构。

示例：

```sql
CREATE INDEX BIndex ON Movie(price);
```

若查询：

```sql
SELECT title, year
FROM Movie, Studio
WHERE price <= 222
  AND Movie.studioName = Studio.name;
```

可先用 `price` 上的辅助索引定位候选记录。

### 3.3 辅助索引中的间接结构

目的：

```text
节约存储空间，避免同一搜索码值对应多个记录时重复保存大量指针。
```

结构：

```text
搜索码 -> 桶/指针列表 -> 数据记录
```

适用：

- 一个搜索码值对应很多记录。
- 需要减少索引项重复。

### 3.4 多索引过滤

当查询条件涉及多个属性时，可分别使用多个索引，得到候选集合后取交集。

例：

```sql
SELECT title
FROM Movie
WHERE studioName = 'Disney'
  AND year = '1995';
```

处理：

```text
1. 用 studioName 索引得到 Disney 的记录集合。
2. 用 year 索引得到 1995 的记录集合。
3. 对两个集合取交集。
4. 读取交集对应的记录。
```

### 3.5 正排索引 vs 倒排索引

文档检索中：

| 类型 | 映射方向 |
| --- | --- |
| 正排索引 | 文档 -> 文档包含哪些词 |
| 倒排索引 | 词 -> 词出现在哪些文档中 |

倒排索引服务于关键词检索。

## 4. B+ 树与变种

### 4.1 B 树、B+ 树、B-link 树

| 类型 | 特点 |
| --- | --- |
| B 树 | 所有节点都可含数据，叶子不一定在同一层 |
| B+ 树 | 仅叶子节点含数据记录指针，叶子节点在同一层 |
| B-link 树 | B+ 树变种，节点带兄弟指针和 high key，便于并发和分裂 |

课件强调：

```text
B+ 树仅叶子节点包含数据，叶子属于同一层。
叶子节点通常按键值顺序通过链指针相连。
```

### 4.2 B+ 树节点结构

中间节点：

```text
n + 1 个指针
n 个键
```

叶节点：

```text
n + 1 个指针
n 个键
其中一个指针通常指向下一个叶节点。
```

搜索效率：

- 节点内键有序。
- 每次比较决定进入哪个子树。
- 树高较低，I/O 次数少。

### 4.3 B+ 树最小填充与最大键数

课件表格口径：

设最大键数为 `n`。

| 节点类型 | 最大指针数 | 最大键数 | 最小指针数 | 最小键数 |
| --- | --- | --- | --- | --- |
| 非根内部节点 | `n + 1` | `n` | `ceil((n+1)/2)` | `ceil((n+1)/2) - 1` |
| 叶节点 | `n + 1` | `n` | `floor((n+1)/2)` | `floor((n+1)/2)` |
| 根节点 | `n + 1` | `n` | 2 | 1 |

注意：

- 对主键上的稠密索引，每条记录通常对应一个键-指针对。
- 主键有序的稀疏索引，每个块可对应一个键-指针对。
- 非键有序属性，常为每个键值一个键-指针对，指向第一个键所在位置或桶。

### 4.4 B+ 树查找模板

等值查询：

```text
1. 从根节点开始。
2. 在节点中找到目标键所在区间。
3. 沿相应指针进入下一层。
4. 重复直到叶节点。
5. 在叶节点中查找目标键，并根据记录指针取数据。
```

范围查询：

```text
1. 先查找范围下界所在叶节点。
2. 在叶节点中顺序扫描。
3. 沿叶节点链表继续向右扫描。
4. 直到超过范围上界。
```

### 4.5 B+ 树插入模板

步骤：

```text
1. 查找应插入的叶节点。
2. 若叶节点有空位，按顺序插入。
3. 若叶节点溢出：
   分裂为两个叶节点；
   调整叶链指针；
   将分裂后的分隔键插入父节点。
4. 若父节点溢出：
   继续分裂并向上提升分隔键。
5. 若根节点溢出：
   创建新的根节点，树高增加 1。
```

考场写法：

```text
插入 key = K：
定位叶节点 L；
L 插入后超过最大键数 n，故分裂；
右侧新叶的最小键 K' 上推到父节点；
若父节点溢出，递归处理。
```

### 4.6 B+ 树删除模板

步骤：

```text
1. 查找包含待删除 key 的叶节点。
2. 删除该 key。
3. 若节点仍满足最小填充，结束。
4. 若节点不足：
   优先向相邻兄弟借键；
   若兄弟不能借，则与兄弟合并。
5. 合并或借用后，调整父节点分隔键。
6. 若父节点不足，递归处理。
7. 若根节点只剩一个子节点，可降低树高。
```

关键操作：

- 兄弟节点合并。
- key 借分。
- 中间节点变根节点。

### 4.7 B+ 树效率

课件思路：

```text
B+ 树查找深度通常很小，I/O 次数约为树高。
插入和删除比查找多一次左右的写入和分裂/合并代价。
```

估算模板：

```text
查找 I/O ≈ 树高 + 读取数据块
插入/删除 I/O ≈ 查找 I/O + 修改叶节点 + 可能的分裂/合并写回
```

## 5. 散列索引

### 5.1 哈希表索引

定义：

```text
用哈希函数 h(key) 将键值映射到桶号。
```

结构：

```text
桶号 -> 桶/桶块 -> 记录或记录指针
```

查询：

```text
1. 计算 h(key)。
2. 直接定位桶。
3. 在桶内扫描。
4. 若有溢出桶，沿桶链继续查找。
```

特点：

- 等值查询快。
- 不适合范围查询。
- 桶太少会溢出，适当时需要增加桶。

课件口令：

```text
每次点查询至少需要一次 I/O。
插入或删除至少需要两次磁盘 I/O。
查询到桶后仍需遍历桶内记录。
桶不是越少越好，合适时应增加。
```

### 5.2 溢出桶与桶链

当桶满时：

```text
新记录可放入溢出桶，并由桶链连接。
```

缺点：

- 桶链过长会降低查询效率。
- 需要适时扩桶或重组。

### 5.3 哈希索引删除

删除方式：

```text
1. 原位删除。
2. 溢出块删除。
3. 桶内整理。
```

若溢出桶变空，可释放或移除。

### 5.4 可扩展散列

核心思想：

```text
使用目录，根据哈希值前缀定位桶；
桶满时按需分裂桶，必要时目录加倍。
```

关键概念：

| 概念 | 含义 |
| --- | --- |
| 全局深度 | 目录使用的哈希前缀位数 |
| 局部深度 | 某个桶实际使用的前缀位数 |
| 目录 | 前缀 -> 桶指针 |
| 桶分裂 | 一个桶分成两个，部分目录指针改向新桶 |

插入模板：

```text
1. 计算 h(key)。
2. 根据全局深度取前缀，定位目录项和桶。
3. 若桶未满，直接插入。
4. 若桶满且局部深度 < 全局深度：
   分裂桶；
   局部深度加 1；
   重分配原桶记录和新记录。
5. 若桶满且局部深度 = 全局深度：
   目录加倍；
   全局深度加 1；
   再执行桶分裂。
```

### 5.5 线性散列

核心思想：

```text
不使用完整目录加倍，而是按分裂指针顺序逐桶分裂。
```

关键变量：

```text
i：当前使用的哈希函数位数
n：当前桶数
r：当前已有记录数
```

特点：

- 桶逐步增长。
- 不需要一次性重新组织全部数据。
- 桶数增长小，数据量增长也可平滑变化。

## 6. 位图索引

### 6.1 位图索引定义

对某个属性的每个可能取值建立一个位向量：

```text
属性值 v -> 位图
```

位图中第 `i` 位表示第 `i` 条记录是否具有该值：

```text
1 表示是，0 表示否。
```

适用：

- 属性取值数量少。
- 多条件查询。
- 数据更新不频繁。

不适用：

- 高基数属性。
- 更新频繁的 OLTP 场景。

### 6.2 位图索引处理等值查询

例：

```sql
SELECT title
FROM Movie
WHERE studioName = 'Disney'
  AND year = 1995;
```

处理：

```text
1. 取 studioName = Disney 的位图。
2. 取 year = 1995 的位图。
3. 两个位图按位 AND。
4. 结果中为 1 的位置对应满足条件的记录。
```

### 6.3 位图索引处理范围查询

例：

```sql
SELECT *
FROM Records
WHERE 45 < age <= 55
  AND 100 < SAL <= 200;
```

模板：

```text
1. 对 age 范围内的取值位图做 OR。
2. 对 SAL 范围内的取值位图做 OR。
3. 两个范围结果再做 AND。
4. 结果位图中为 1 的记录即查询结果。
```

课件示例口令：

```text
范围内部使用 OR，不同条件之间使用 AND。
```

### 6.4 压缩位图索引

目的：

```text
减少位图存储空间，提高存储和扫描效率。
```

思路：

- 对连续 0 或连续 1 的片段压缩编码。
- 查询时可在压缩编码上直接执行与/或操作。

管理操作：

- 插入记录：生成新记录号，分配新位置。
- 删除记录：可标记删除。
- 修改索引值：更新对应位置的位。

## 7. 倒排索引

### 7.1 倒排索引定义

对每个词项 `t`，存储包含该词的文档列表：

```text
term -> postings list(docID...)
```

也可保存：

- 词频。
- 词在文档中的位置。
- 文档权重。

### 7.2 为什么需要位置列表

如果只存文档 ID 列表：

```text
可回答“哪些文档含某词”。
```

如果要回答短语查询或邻近查询：

```text
需要位置列表 postings lists。
```

### 7.3 倒排索引构建流程

课件流程：

```text
1. 分词：把文档字符串切分成词。
2. 规范化：统一大小写、词形等。
3. 词干提取：提取词干或词根。
4. 去停用词：去掉 the、a、to、of 等常见词。
5. 得到 Token-docID 序列。
6. 根据 token 排序，docID 作为第二排序键。
7. 建立词典和位置列表。
8. 合并同一词在同一文档中的多次出现，记录频率和位置。
```

### 7.4 AND 查询处理

例：

```text
Brutus AND Caesar
```

模板：

```text
1. 在词典中定位 Brutus，得到位置列表 L1。
2. 在词典中定位 Caesar，得到位置列表 L2。
3. 合并两个有序位置列表，取 docID 交集。
4. 交集中的文档就是查询结果。
```

若列表有序，双指针合并即可：

```text
若 docID1 = docID2，输出并同时后移；
若 docID1 < docID2，移动 L1；
若 docID1 > docID2，移动 L2。
```

### 7.5 存储与消耗

倒排索引主要存：

- 词典。
- 词频。
- docID 列表。
- 位置列表。

空间代价与文档数量、词项数量、词频和是否保存位置信息有关。

## 8. 高维索引

### 8.1 SQL 中的多维查询

多维数据经常用于传统关系数据库中的非传统查询：

- 部分匹配查询。
- 范围查询。
- 最近邻查询。
- Where-I-am 查询。

例：

```sql
SELECT id
FROM Rectangle
WHERE xll <= 10 AND yll <= 20
  AND xur >= 10 AND yur >= 20;
```

最近邻查询常写成：

```sql
SELECT *
FROM Point p
WHERE NOT EXISTS (
  SELECT *
  FROM Point q
  WHERE (q.x - 10)*(q.x - 10) + (q.y - 20)*(q.y - 20)
      < (p.x - 10)*(p.x - 10) + (p.y - 20)*(p.y - 20)
);
```

### 8.2 为何不能只用普通 B 树

多维点查询如果只用单属性 B 树：

```text
可能只能在一个维度上快速过滤，另一个维度仍需大量扫描。
```

常见改进：

- 分段散列函数。
- 网格文件。
- 多维树结构。

### 8.3 网格文件

思想：

```text
将多维空间划分为网格，每个网格单元指向包含相应点的桶。
```

适用：

- 点数据。
- 多属性等值或范围查询。

问题：

- 空间划分可能产生空单元。
- 维度高时目录可能膨胀。

### 8.4 分段散列函数

思想：

```text
将多个键的哈希结果分段组合，形成多维索引入口。
```

优点：

- 可用于多维等值查询。

局限：

- 对范围查询支持较弱。

### 8.5 多维树结构

课件列出：

| 结构 | 适合数据 |
| --- | --- |
| 多键索引 | 多属性组合查询 |
| kd 树 | 点数据 |
| 四叉树 | 点数据或空间递归划分 |
| R 树 | 区域/矩形对象 |

### 8.6 多键索引性能

部分匹配：

- 若第一个属性被指定，访问效率高。
- 若第一个属性没有指定，可能需要搜索子索引，开销较大。

范围查询：

- 若各属性都支持范围查询，效率较高。
- 先在第一属性索引上范围查询，再逐级进入子索引。

最近邻查询：

- 可通过一系列范围查询完成。
- 查询范围逐步扩大，直到找到最近点。

### 8.7 kd 树

特点：

- 二叉树，内部节点按某一维划分空间。
- 内部节点存一个键和两个子节点指针。
- 所有相同层级节点可能按照同一维划分。
- 叶子节点存记录指针。

复杂查询思路：

```text
从根开始按维度比较；
若查询区域与另一侧空间也相交，需要回溯搜索另一侧。
```

### 8.8 四叉树

特点：

- 每个内部节点把二维空间划分成四个象限。
- 适合二维点或区域数据。
- 若一个正方形中的点数超过阈值，则递归分裂。

### 8.9 R 树

适合：

```text
区域对象、矩形对象、空间对象。
```

特点：

- 内部节点对应某个内部区域。
- 子区域不一定互斥，可能重叠。
- 查询区域与子节点区域相交时，需要访问该子节点。

Where-am-I 查询：

```text
从根节点开始；
检查当前节点的子区域；
若查询点或区域与子区域相交，则递归进入；
若有多个相交子区域，可能需要访问多个分支。
```

## 9. LSM Tree

### 9.1 动机

适用：

```text
更新密集、写密集场景。
```

课件例子：

- 微博、微信：数据追加写入，不允许修改。

B+ 树问题：

```text
在更新密集场景会产生大量随机 I/O。
```

LSM 思想：

```text
将更新操作转化为批量顺序写入。
多级索引积累更新后进行索引合并。
索引合并可通过顺序写入实现。
```

### 9.2 二阶 LSM 树

结构：

```text
C0：内存索引，如红黑树、AVL 树。
C1：磁盘索引，如 B+ 树。
```

插入：

```text
1. 先插入内存 C0。
2. 当 C0 达到设定阈值，滚动合并到 C1。
```

查找：

```text
1. 先在 C0 查找。
2. 若找到，返回。
3. 若未找到，再到 C1 查找。
```

删除：

```text
使用删除标记。
若在 C0 中存在对应条目，则从 C0 删除；
若不在 C0，则插入删除条目；
合并到 C1 时再真正处理。
```

### 9.3 LSM 滚动合并

合并条件：

```text
C0 树容量达到设定阈值。
```

滚动合并过程：

```text
1. 在 C1 中找到与 C0 数据相邻或对应的连续页面。
2. 从 C0 取出节点，与磁盘块进行合并。
3. 空页块中的旧 C1 数据被移除。
4. 将合并后的新 C1 块写回磁盘。
```

### 9.4 多阶 LSM 树

目的：

```text
降低相邻层级滚动合并参数，引入磁盘上的多级索引。
```

结论：

```text
相邻层级容量比例固定时，LSM 树 I/O 成本最低。
```

### 9.5 LSM 优化

课件列出：

```text
1. 滚动合并实现复杂，需增加简化条件。
2. LSM 树性能问题：
   读放大：查找时可能要查多层索引。
   写放大：合并需要多次写磁盘。
   空间放大：旧值和删除标记可能暂存多层。
3. 可通过布隆过滤器、分区优化、异步压缩机制优化。
```

### 9.6 LSM 与 B+ 树对比

| 操作 | B+ 树 | LSM 树 |
| --- | --- | --- |
| 插入 | `O(log n)` | `O(1)` |
| 删除 | `O(log n)` | `O(1)` |
| 更新 | `O(log n)` | `O(1)` |
| 查询 | `O(log n)` | `O(L * log n)` |

其中：

```text
L 为 LSM 层数，n 为记录数。
```

适用场景：

- B+ 树：读写均衡，读密集负载。
- LSM 树：连续写入，写密集负载。

## 10. 高频题型模板

### 10.1 “选择合适索引”

答题模板：

```text
1. 等值查询且不需要范围：可选哈希索引。
2. 范围查询、排序、前缀匹配：优先 B+ 树。
3. 多条件低基数属性：可选位图索引。
4. 文本关键词查询：使用倒排索引。
5. 空间/多维范围查询：使用 kd 树、四叉树或 R 树。
6. 写密集、追加写入：使用 LSM Tree。
```

### 10.2 “稠密索引与稀疏索引比较”

答题模板：

```text
稠密索引为每个搜索码值或记录建立索引项，查找直接，但空间和维护代价大。
稀疏索引只为部分搜索码值建立索引项，通常每个数据块一个索引项，空间小，但要求数据文件有序，查找后还要在块内扫描。
```

### 10.3 “B+ 树插入过程”

答题模板：

```text
插入 key = K：
1. 从根节点查找到应插入的叶节点。
2. 若叶节点未满，按顺序插入 K。
3. 若叶节点满，分裂叶节点，调整叶链。
4. 将新右叶的最小键插入父节点。
5. 若父节点也溢出，继续分裂并上推。
6. 若根节点溢出，创建新根，树高加 1。
```

### 10.4 “B+ 树删除过程”

答题模板：

```text
删除 key = K：
1. 查找到包含 K 的叶节点并删除。
2. 若删除后节点仍满足最小填充，结束。
3. 若不足，先向兄弟节点借键。
4. 若不能借，则与兄弟节点合并。
5. 调整父节点分隔键。
6. 若父节点不足，递归处理。
7. 若根节点只剩一个孩子，降低树高。
```

### 10.5 “哈希索引插入/查找”

答题模板：

```text
查找 key：
1. 计算桶号 b = h(key)。
2. 访问桶 b。
3. 在桶内顺序查找。
4. 若有溢出桶，沿桶链继续查找。

插入 key：
1. 计算 h(key)。
2. 若桶有空位，直接插入。
3. 若桶满，放入溢出桶或触发扩桶。
```

### 10.6 “可扩展散列插入”

答题模板：

```text
1. 根据全局深度 d 取哈希前 d 位定位目录。
2. 若目标桶未满，插入。
3. 若目标桶满且局部深度 < 全局深度：
   分裂桶，局部深度加 1，重分配记录。
4. 若目标桶满且局部深度 = 全局深度：
   目录加倍，全局深度加 1，再分裂桶。
```

### 10.7 “位图索引计算”

答题模板：

```text
等值条件：直接取该值的位图。
同一属性多个取值或范围：位图之间做 OR。
不同属性条件同时满足：位图之间做 AND。
最终结果位图中为 1 的位置就是满足条件的记录。
```

### 10.8 “倒排索引 AND 查询”

答题模板：

```text
1. 在词典中查找每个关键词。
2. 取出对应 postings list。
3. 对 postings list 按 docID 取交集。
4. 交集中的 docID 即满足 AND 查询的文档。
```

### 10.9 “高维索引选择”

答题模板：

```text
点数据：可选 kd 树、四叉树或网格文件。
区域/矩形对象：优先 R 树。
多属性组合查询：可用多键索引。
范围查询：选择能剪枝空间区域的结构。
最近邻查询：先设定查询范围，必要时逐步扩大范围。
```

### 10.10 “LSM 与 B+ 树比较”

答题模板：

```text
B+ 树所有页面组成一个树状索引，查找效率稳定，但更新会产生随机 I/O，适合读密集或读写均衡负载。
LSM 树将写入先放入内存层，再批量顺序合并到磁盘层，写入效率高，但查询可能需要查多层索引，适合写密集负载。
```

## 11. 易错清单

- 稀疏索引通常要求数据文件按搜索码有序；辅助索引通常更接近稠密索引。
- 多级索引是“对索引再建索引”，不是对数据文件建多个不同索引。
- B+ 树数据记录指针主要在叶节点；内部节点只用于导航。
- B+ 树范围查询依赖叶节点链表，不是每次都从根重新查。
- B+ 树插入分裂时，上推的是分隔键；删除时要注意父节点分隔键调整。
- 哈希索引适合等值查询，不适合范围查询。
- 桶越少越容易溢出，桶链太长会退化为扫描。
- 可扩展散列要区分全局深度和局部深度。
- 位图索引适合低基数、读多写少场景；高基数或频繁更新不合适。
- 位图范围查询通常先 OR 同一属性范围，再 AND 不同属性条件。
- 倒排索引不是“文档到词”，而是“词到文档列表”。
- R 树子区域可能重叠，查询可能访问多个分支。
- LSM Tree 写快，但读可能放大，需要合并、布隆过滤器等优化。
