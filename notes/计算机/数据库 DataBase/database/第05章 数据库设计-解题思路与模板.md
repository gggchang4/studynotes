# 第05章 数据库设计：开卷考试解题思路与模板

> 适用范围：数据库设计步骤、需求分析、数据流图 DFD、数据字典 DD、概念结构设计、E-R 图、逻辑结构设计、E-R 模型到关系模型转换、关系模式优化、物理设计、数据库实施与维护。

## 0. 考场快速判题地图

| 题目关键词 | 题型 | 先套什么模板 |
| --- | --- | --- |
| 数据库设计分哪几个阶段 | 概念题 | 六阶段：需求、概念、逻辑、物理、实施、运行维护 |
| 需求分析任务/方法/成果 | 简答题 | 收集需求、分析需求、形成数据字典和数据流图 |
| 画数据流图 DFD | 图形题 | 外部实体、处理、数据流、数据存储 |
| 写数据字典 | 描述题 | 数据项、数据结构、数据流、数据存储、处理过程 |
| 画 E-R 图 | 设计题 | 实体、属性、联系、基数、弱实体/多值属性 |
| 合并局部 E-R 图 | 设计题 | 消除属性冲突、命名冲突、结构冲突 |
| E-R 转关系模式 | 计算/设计题 | 实体转表，联系按 1:1、1:n、m:n 处理 |
| 关系模式优化 | 简答/设计题 | 分析依赖、极小化、判范式、合并/分解 |
| 物理设计 | 简答题 | 存储结构、索引、聚簇、系统参数 |
| 数据库实施和维护 | 简答题 | 建库、装入数据、编程测试、备份恢复、安全完整性 |

**数据库设计大题通用答题顺序：**

1. 明确业务对象和处理需求。
2. 写出需求分析成果：数据流图、数据字典。
3. 抽取实体、属性、联系，画局部或全局 E-R 图。
4. 将 E-R 图转换为关系模式，标出主键、外键。
5. 对关系模式进行规范化和优化。
6. 说明物理设计选择：索引、聚簇、存储位置。
7. 说明实施与维护：建表、装入、测试、备份、安全、性能调整。

## 1. 数据库设计概述

### 1.1 数据库设计的含义

数据库设计是在给定应用环境下，根据数据模型构造最优数据库模式，建立数据库及其应用系统，使其能够有效存储数据，并满足用户应用需求。

数据库设计包括：

- 结构特性设计：设计数据库结构、模式、表、关系。
- 行为特性设计：设计数据处理、应用程序、事务流程。

### 1.2 数据库设计目标

- 满足用户需求。
- 模型结构清晰、语义准确。
- 数据冗余较小，完整性好。
- 查询、更新、维护效率较高。
- 数据库安全、可靠、易扩展。

### 1.3 数据库设计特点

- 数据库建设是硬件、软件和人员管理的结合。
- 数据库设计应与应用系统设计结合。
- 设计过程要把结构设计和行为设计结合起来。

### 1.4 数据库设计六阶段

| 阶段 | 任务 | 主要成果 |
| --- | --- | --- |
| 1. 需求分析 | 收集、分析用户需求 | 数据流图、数据字典 |
| 2. 概念结构设计 | 抽象现实世界，形成独立于 DBMS 的概念模型 | E-R 图 |
| 3. 逻辑结构设计 | 将概念模型转换为 DBMS 支持的数据模型并优化 | 关系模式、外模式 |
| 4. 物理设计 | 为逻辑模型选择物理结构和存取方法 | 存储结构、索引、聚簇 |
| 5. 数据库实施 | 建库、编程、装入数据、试运行 | 数据库和应用程序 |
| 6. 运行和维护 | 正式运行后评价、调整、修改 | 备份恢复、安全维护、性能优化 |

记忆口诀：

```text
需求 -> 概念 -> 逻辑 -> 物理 -> 实施 -> 运行维护
```

## 2. 需求分析

### 2.1 需求分析的任务

需求分析通过详细调查现实世界要处理的对象，充分了解原系统工作概况，明确用户各种需求，并在此基础上确定新系统功能。

重点：

- 调查用户的数据需求：需要存储哪些数据，数据之间有什么联系。
- 调查用户的处理需求：需要完成哪些功能，响应时间、处理方式有什么要求。
- 调查安全性和完整性需求：哪些用户能访问哪些数据，需要哪些约束。
- 得到需求分析说明书、数据流图、数据字典等成果。

### 2.2 需求信息收集内容

```text
1. 了解组织机构设置。
2. 了解主要业务活动和职能。
3. 了解系统中各种外部要求。
4. 确定应用系统边界。
```

### 2.3 常用调查方法

| 方法 | 说明 |
| --- | --- |
| 跟班作业 | 观察用户实际工作流程 |
| 开调查会 | 集中了解业务和问题 |
| 请专人介绍 | 由熟悉业务的人讲解 |
| 询问 | 对关键用户逐项提问 |
| 设计调查表请用户填写 | 批量收集标准化需求 |
| 查阅记录 | 查已有表单、报表、档案、规章 |

### 2.4 需求分析方法

| 方法 | 用法 |
| --- | --- |
| 自顶向下 | 先分析总体目标，再逐层分解 |
| 自底向上 | 从具体业务和数据出发，逐步归纳 |
| 结构化分析 SA | 用数据流图和数据字典描述系统 |

## 3. 数据流图 DFD

### 3.1 DFD 的作用

数据流图用于描述业务流程中数据的流动、处理和存储，展示系统中各处理过程之间以及系统与外部实体之间的数据联系。

### 3.2 DFD 基本符号

| 符号 | 含义 | 说明 |
| --- | --- | --- |
| 外部实体 | 数据的来源或去向 | 如学生、供应商、教师 |
| 处理过程 | 对数据进行加工处理 | 如注册、选课、审核采购计划 |
| 数据流 | 数据从一个对象流向另一个对象 | 如通知单、订单、成绩单 |
| 数据存储 | 数据暂存或长期保存位置 | 如学生数据库、采购记录 |

### 3.3 画 DFD 的步骤模板

```text
1. 确定系统边界。
2. 找外部实体：谁向系统提供数据，谁接收系统输出。
3. 找主要处理：系统要完成哪些业务动作。
4. 找数据流：每个处理的输入和输出是什么。
5. 找数据存储：哪些数据需要保存。
6. 从顶层图开始，逐层细化为 0 层、1 层等子图。
7. 检查输入输出是否平衡。
```

### 3.4 DFD 检查清单

- 每个处理至少有输入和输出。
- 数据流名称应是名词或名词短语。
- 处理名称应是动宾结构，如“审核计划”“生成订单”。
- 数据存储应有读写数据流。
- 上层图与下层图的数据流要平衡。
- 不要把控制流、物资流当成数据流。

## 4. 数据字典 DD

### 4.1 数据字典作用

数据字典是系统中各种数据描述的集合，对数据流图中的数据项、数据结构、数据流、数据存储、处理过程进行详细说明。

作用：

- 对 DFD 中的数据进行解释。
- 帮助用户和设计人员统一术语。
- 是后续概念设计、逻辑设计和物理设计的重要依据。
- 在系统运行过程中帮助 DBA 掌握系统情况。

### 4.2 数据字典内容

数据字典通常包括：

```text
1. 数据项
2. 数据结构
3. 数据流
4. 数据存储
5. 处理过程
```

### 4.3 数据项描述模板

数据项是不可再分的数据单位。

```text
数据项名：
含义说明：
别名：
数据类型：
长度：
取值范围：
是否允许空值：
与其他数据项的逻辑关系：
```

例：

```text
数据项名：订单编号
类型：CHAR
长度：8
别名：采购单号
取值范围：10000000-99999999
```

### 4.4 数据结构描述模板

数据结构是若干数据项有意义的集合，反映数据之间的组合关系。

```text
数据结构名：
含义说明：
组成：{数据项或数据结构}
```

例：

```text
数据结构名：采购细节
含义说明：作为采购计划的组成部分，说明对某个产品要采购哪些零件、每种零件采购多少数量。
组成：零件号、零件名、数量
```

### 4.5 数据流描述模板

数据流表示加工处理过程的输入或输出数据。

```text
数据流名：
说明：
来源：
去向：
组成：
平均流量：
高峰期流量：
```

例：

```text
数据流名：采购计划
说明：由各产品所需零件数、选定供应商和审核情况形成采购计划。
来源：确定采购计划
去向：编制订单
组成：采购细节、采购审核
```

### 4.6 数据存储描述模板

数据存储是处理过程中数据的存取位置。

```text
数据存储名：
说明：
编号：
流入的数据流：
流出的数据流：
组成：
数据量：
存取方式：
```

例：

```text
数据存储名：产品
说明：对每种产品的品名、规格的描述，并对每种产品做成本预算。
流入数据流：预算
流出数据流：产品号、产品名、预算
存取方式：随机存取
```

### 4.7 处理过程描述模板

```text
处理过程名：
说明：
输入：
输出：
处理逻辑：
```

例：

```text
处理过程名：确定采购计划
说明：对要采购的每一零件，根据零件库存量确定采购数量，再根据供应商报价选择合适供应商，制定采购计划。
输入：供应商报价、零件库存、已批准生产计划
输出：采购计划
处理逻辑：
1. 对每种采购零件计算供应商报价。
2. 选择报价最低的供应商。
3. 将供应商写入采购清单。
```

## 5. 概念结构设计

### 5.1 概念结构设计任务

将需求分析得到的用户需求抽象为信息结构，形成独立于具体 DBMS 的概念模型，通常用 E-R 图表示。

概念模型要求：

- 能真实、充分反映现实世界。
- 易于理解。
- 易于更改。
- 易于向关系、网状、层次等数据模型转换。

### 5.2 概念结构设计方法

| 方法 | 思路 |
| --- | --- |
| 自顶向下 | 先定义全局概念框架，再逐步细化 |
| 自底向上 | 先设计局部 E-R 图，再集成为全局 E-R 图 |
| 逐步扩张 | 先定义核心概念结构，再向外扩充 |
| 混合策略 | 自顶向下设计全局框架，自底向上集成局部视图 |

### 5.3 自底向上设计步骤

```text
1. 数据抽象与局部视图设计。
2. 将各局部视图集成为全局 E-R 图。
```

局部 E-R 图设计：

```text
1. 选择局部应用。
2. 逐一设计 E-R 图。
3. 实体与属性划分原则：
   属性应是不可再分的数据项。
   属性不能再具有需要描述的性质。
   属性不应与其他实体有联系。
```

### 5.4 E-R 图基本元素

| 元素 | 图形习惯 | 说明 |
| --- | --- | --- |
| 实体型 | 矩形 | 如学生、课程、教师、产品、零件 |
| 属性 | 椭圆 | 如学号、姓名、价格 |
| 联系 | 菱形 | 如选修、供应、组成 |
| 基数 | `1`、`n`、`m` 或范围 | 表示实体之间参与联系的数量约束 |

### 5.5 E-R 扩展元素

| 元素 | 说明 |
| --- | --- |
| 聚集 | 把某个联系及相关实体看作更高层实体 |
| 概括/泛化 | 超类与子类关系，如职工分为飞行员、机械师、管理人员 |
| 单值属性 | 一个实体实例只有一个属性值 |
| 多值属性 | 一个实体实例可能有多个属性值，如电话号码 |
| 派生属性 | 可由其他属性计算得到，如年龄可由出生日期得到 |
| 复合属性 | 可再分的属性，如地址可拆为省、市、街道等 |
| 弱实体 | 依赖其他实体存在的实体 |

### 5.6 联系基数判定

| 联系类型 | 判断口令 | 例子 |
| --- | --- | --- |
| 1:1 | A 的一个实例最多对应 B 的一个实例，反之也一样 | 班级与班长 |
| 1:n | A 的一个实例对应多个 B，B 的一个实例只对应一个 A | 部门与职工 |
| m:n | A 的一个实例可对应多个 B，B 的一个实例也可对应多个 A | 学生与课程 |

最小/最大基数示例：

```text
学生 1..6 选课 0..40 课程
```

表示一个学生选 1 到 6 门课，一门课程可被 0 到 40 个学生选修。

### 5.7 合并局部 E-R 图

局部 E-R 图合并是发现并解决冲突的过程。

常见冲突：

| 冲突 | 表现 | 处理方法 |
| --- | --- | --- |
| 属性冲突 | 同一属性取值类型、范围、单位不同 | 统一类型、单位、范围 |
| 命名冲突 | 同名异义或异名同义 | 重命名，统一术语 |
| 结构冲突 | 同一对象在不同图中作为实体/属性/联系不同 | 统一为合适结构 |

合并后要消除冗余：

- 分析法：根据语义判断冗余数据和冗余联系。
- 规范化法：用函数依赖和最小依赖集判断冗余联系。

## 6. 逻辑结构设计

### 6.1 逻辑结构设计任务

将基本 E-R 模型转换为具体 DBMS 支持的数据模型，例如关系模型，并进行优化。

步骤：

```text
1. 将概念结构转换为关系模型。
2. 优化数据模型。
3. 设计适合 DBMS 的子模式/外模式。
```

### 6.2 E-R 模型转关系模型总规则

课件给出的转换原则：

```text
1. 一个实体型转换为一个关系模式。
2. 一个 1:1 联系可转换为一个独立关系模式，也可与任一端实体关系模式合并。
3. 一个 1:n 联系可转换为一个独立关系模式，也可与 n 端对应关系模式合并。
4. 一个 m:n 联系转换为一个关系模式。
5. 三个或三个以上实体间的多元联系转换为一个关系模式。
6. 相同键的关系模式可以合并。
```

### 6.3 实体转换模板

实体型：

```text
实体名(属性1, 属性2, ..., 标识属性)
```

转换为：

```text
实体名(标识属性, 其他属性...)
主键：标识属性
```

例：

```text
产品(P#, PP, PC)
零件(S#, SP)
材料(M#, MC)
仓库(L#, LC)
```

### 6.4 1:1 联系转换

方法一：转换为独立关系模式。

```text
联系名(A主键, B主键, 联系属性...)
主键：A主键 或 B主键
外键：A主键 引用 A，B主键 引用 B
```

方法二：合并到任一端实体。

```text
A(A主键, A属性..., B主键, 联系属性...)
```

选择建议：

- 若联系有较多属性，单独建表更清晰。
- 若联系非常稳定且常与某一端一起查询，可合并。

### 6.5 1:n 联系转换

方法一：合并到 n 端实体关系模式。

```text
n端实体(n端主键, n端属性..., 1端主键, 联系属性...)
外键：1端主键 引用 1端实体
```

方法二：独立关系模式。

```text
联系名(1端主键, n端主键, 联系属性...)
主键：n端主键 或 组合键
```

常用答案：

> 1:n 联系通常把 1 端主键加入 n 端关系模式，并把联系属性也加入 n 端。

例：职工与家属。

```text
职工(工号, 姓名, 年龄, 性别, 职务)
家属(工号, 家属姓名, 家属关系)
```

其中 `工号` 是家属关系中的外键。

### 6.6 m:n 联系转换

m:n 联系必须转换为独立关系模式。

```text
联系名(A主键, B主键, 联系属性...)
主键：(A主键, B主键)
外键：A主键 引用 A，B主键 引用 B
```

例：学生选课。

```text
学生(学号, 姓名, ...)
课程(课程号, 课程名, ...)
选课(学号, 课程号, 成绩)
主键：(学号, 课程号)
```

### 6.7 多元联系转换

三个或三个以上实体之间的联系转换为一个独立关系模式：

```text
联系名(A主键, B主键, C主键, 联系属性...)
主键：通常为参与实体主键的组合
外键：分别引用各实体
```

### 6.8 弱实体转换

弱实体依赖强实体存在。

```text
强实体(强实体主键, ...)
弱实体(强实体主键, 弱实体部分键, 弱实体属性...)
主键：(强实体主键, 弱实体部分键)
外键：强实体主键 引用 强实体
```

例：

```text
职工(工号, 姓名, 年龄, 性别, 职务)
家属(工号, 家属姓名, 家属关系)
主键：(工号, 家属姓名)
```

### 6.9 自联系转换

#### 1:n 自联系

把同一实体的主键作为外键加入该实体关系中，并改名区分角色。

例：职工领导职工。

```text
职工(工号, 姓名, 年龄, 性别, 职称, 领导工号, 民意测验)
主键：工号
外键：领导工号 引用 职工(工号)
```

#### m:n 自联系

转换为独立关系模式，两个外键都引用同一实体。

例：零件组成零件。

```text
零件(代码, 名称, 价格)
组成(父零件代码, 子零件代码, 数量)
主键：(父零件代码, 子零件代码)
外键：父零件代码、子零件代码 均引用 零件(代码)
```

### 6.10 多值属性转换

多值属性单独转换为关系模式：

```text
实体名(实体主键, 其他单值属性...)
实体_多值属性(实体主键, 多值属性)
主键：(实体主键, 多值属性)
外键：实体主键 引用 实体名
```

例：

```text
职工(职工号, 姓名, 出生日期, 省, 市, 邮政编码, 街道号, 街道名, 楼栋号)
职工电话(职工号, 电话号码)
```

### 6.11 超类/子类转换

常见做法：

```text
超类(超类主键, 公共属性...)
子类1(超类主键, 子类特有属性...)
子类2(超类主键, 子类特有属性...)
```

例：

```text
职工(职工号, 姓名, 性别, 年龄, 参加工作时间)
飞行员(职工号, 飞行时间, 健康检查, 飞机型号)
机械师(职工号, 学历, 级别, 专业职称)
管理人员(职工号, 职务, 职称)
```

说明：

> 可在超类中增加一个指示器属性，根据指示器值查询子类实体。

### 6.12 关系模式优化

优化步骤：

```text
1. 分析数据依赖。
2. 对数据依赖进行极小化处理，消除冗余联系。
3. 确定各关系模式的范式。
4. 根据处理要求，对某些模式进行合并或分解。
5. 为提高效率和利用率，对关系模式进一步分解。
```

常用分解方法：

- 垂直分解：按属性组拆分，一个关系拆成多个列集合。
- 水平分解：按元组条件拆分，一个关系拆成多个行集合。

设计用户子模式时考虑：

- 使用安全。
- 使用方便。
- 符合用户习惯。

## 7. 物理设计

### 7.1 物理设计任务

为给定逻辑模型选择最适合应用环境的物理结构，包括存储结构和存取方法。

步骤：

```text
1. 确定数据库的物理结构。
2. 对物理结构进行评价。
3. 必要时调整逻辑设计或物理设计。
```

物理设计目标：

- 提高访问效率。
- 降低存储空间。
- 减少维护代价。
- 保证安全性和可靠性。

### 7.2 存储结构设计

确定数据存放位置：

- 经常一起访问的数据尽量存放在同一磁盘或相邻位置。
- 访问频繁的数据为热点数据，可单独优化。
- 重要数据、日志、备份可以放在不同存储介质上。

确定系统配置参数：

- 用户数。
- 缓冲区大小。
- 时间片。
- 数据库大小。
- 对象数。
- 装填因子等。

### 7.3 是否建立索引

建立索引要考虑：

```text
1. 是否建索引？
2. 在哪些属性上建索引？
3. 是否建聚簇索引？
4. 索引维护代价。
```

索引目的：

- 提供多种存取路径。
- 加快查询速度。

不适合建立索引的情况：

```text
1. 太小的表。
2. 很少出现在查询条件中的属性。
3. 经常更新的属性和表。
4. 经常查询大部分记录的属性和表。
5. 属性值取值个数很少的属性，如性别。
6. 属性值分布严重不均的属性，如年龄。
```

### 7.4 适合建立索引的情况

```text
1. 主键通常建立索引。
2. 外键通常建立索引。
3. 经常出现在 WHERE 条件中的属性。
4. 经常用于连接操作的属性。
5. 经常出现在 ORDER BY、GROUP BY、DISTINCT 中的属性。
6. 经常进行范围查询或最大/最小值查询的属性。
```

### 7.5 聚簇设计

聚簇是把相关数据按物理存储顺序集中存放，提高特定访问效率的技术。

适合建立聚簇：

- 经常按某个属性顺序访问记录。
- 某些关系经常一起连接查询。
- 希望减少磁盘 I/O。

不适合建立过多聚簇：

- 维护代价高。
- 更新频繁会影响性能。
- 一个表通常只能有有限的聚簇顺序。

建立聚簇的一般原则：

```text
1. 通过聚簇访问或连接的应用多，与聚簇无关的访问少。
2. 对应每个聚簇码值的平均元组数不太少。
3. 聚簇码值相对稳定，避免频繁修改。
```

## 8. 数据库实施与维护

### 8.1 数据库实施主要任务

```text
1. 定义数据库结构。
2. 组织数据入库。
3. 编制与调试应用程序。
4. 数据库试运行。
```

数据入库常见步骤：

```text
1. 筛选数据。
2. 转换数据格式。
3. 输入数据。
4. 校验数据。
```

测试：

- 功能测试：检查应用程序能否完成各种操作。
- 性能测试：检查性能指标是否达到设计目标。

### 8.2 数据库维护内容

数据库运行后维护主要由 DBA 完成，包括：

```text
1. 数据库转储与恢复。
2. 数据库安全性、完整性控制。
3. 数据库性能监督、分析和提升。
4. 数据库重组织与重构造。
```

### 8.3 转储与恢复

目的：

- 定期备份数据库和日志文件。
- 当系统故障时能恢复到一致状态。
- 尽量减少故障造成的数据丢失。

### 8.4 安全性与完整性控制

维护内容：

- 根据用户职责调整访问权限。
- 根据业务变化调整完整性约束。
- 定期审计异常访问。
- 防止非法访问和错误操作。

### 8.5 性能监督与调整

DBMS 通常提供性能监控工具，DBA 可利用这些工具获得系统运行情况，并通过调整来改进性能。

可能调整：

- 索引。
- 聚簇。
- 缓冲区和系统参数。
- 查询语句。
- 关系模式或存储结构。

### 8.6 重组织与重构造

| 操作 | 含义 |
| --- | --- |
| 重组织 | 不改变数据库逻辑结构，只改善物理存储，提高效率 |
| 重构造 | 当应用需求变化较大时，修改数据库模式或结构 |

## 9. 高频设计题模板

### 9.1 “简述数据库设计步骤”

答题模板：

```text
数据库设计一般包括六个阶段：
1. 需求分析：收集和分析用户需求，形成数据字典和数据流图。
2. 概念结构设计：抽象现实世界，形成独立于 DBMS 的 E-R 模型。
3. 逻辑结构设计：将 E-R 模型转换为关系模型，并进行规范化和优化。
4. 物理设计：为逻辑模型选择存储结构和存取方法，如索引、聚簇。
5. 数据库实施：用 DBMS 建库、装入数据、编写和调试应用程序。
6. 运行和维护：数据库正式运行后进行备份恢复、安全完整性控制、性能优化和结构调整。
```

### 9.2 “画 E-R 图”

答题步骤：

```text
1. 从题干名词中找实体：学生、课程、教师、产品、零件等。
2. 从描述性信息中找属性：编号、姓名、价格、数量等。
3. 找实体之间的动词或业务关系：选修、供应、组成、管理等。
4. 判断联系基数：1:1、1:n、m:n。
5. 判断联系是否有属性：成绩、数量、日期等。
6. 判断是否存在弱实体、多值属性、复合属性、派生属性。
7. 画图并标出主标识属性和联系基数。
```

检查句：

```text
实体不能遗漏，联系属性不能放错，m:n 联系通常要保留联系属性。
```

### 9.3 “将 E-R 图转换为关系模式”

答题步骤：

```text
1. 每个实体先转为一个关系模式。
2. 处理 1:1 联系：可独立成表，也可合并到任一端。
3. 处理 1:n 联系：通常把 1 端主键加入 n 端。
4. 处理 m:n 联系：单独成表，主键为两端主键组合，加联系属性。
5. 处理多元联系：单独成表，主键通常为各实体主键组合。
6. 处理弱实体：加入强实体主键，与弱实体部分键组成主键。
7. 处理多值属性：单独成表。
8. 标出每个关系的主键和外键。
9. 合并相同键的关系模式。
```

输出格式：

```text
实体A(A#, A1, A2)
实体B(B#, B1, B2)
联系R(A#, B#, R属性)
主键：(A#, B#)
外键：A# 引用 实体A，B# 引用 实体B
```

### 9.4 “给定业务，设计关系模式”

模板：

```text
1. 识别实体：
   E1(...)
   E2(...)

2. 识别联系：
   E1 与 E2 是 1:n / m:n / 1:1。

3. 转换关系：
   E1(E1主键, 属性...)
   E2(E2主键, 属性...)
   若 m:n，则 R(E1主键, E2主键, 联系属性...)

4. 标注完整性：
   主键：...
   外键：...
   非空/唯一/检查约束：...

5. 简要说明优化：
   消除冗余，满足适当范式；常查属性可建索引。
```

### 9.5 “写数据字典条目”

如果题目要求写某个数据项：

```text

```

如果题目要求写某个数据流：

```text

```

如果题目要求写处理过程：

```text

```

### 9.6 “关系模式优化”

答题模板：

```text
1. 分析函数依赖，找出候选键。
2. 求最小函数依赖集，消除冗余依赖。
3. 判断各关系模式范式。
4. 若存在插入、删除、修改异常，则分解到 3NF 或 BCNF。
5. 若查询经常需要连接且冗余可接受，可适当合并。
6. 若关系过大，可根据处理要求垂直分解或水平分解。
```

### 9.7 “是否应该建立索引”

答题模板：

```text
适合建索引：
主键、外键、经常用于 WHERE、JOIN、ORDER BY、GROUP BY、DISTINCT 的属性，
或经常进行范围查询、最大最小值查询的属性。

不适合建索引：
小表、很少查询的属性、频繁更新的属性、取值很少或分布严重不均的属性，
以及经常需要扫描大部分记录的表。
```

### 9.8 “数据库维护包括什么”

答题模板：

```text
数据库维护包括：
1. 数据库转储与恢复，保障故障后恢复。
2. 安全性和完整性控制，根据用户和业务变化调整权限和约束。
3. 性能监督、分析和改进，如调整索引、聚簇和参数。
4. 数据库重组织和重构造，使数据库适应数据增长和需求变化。
```

## 10. 易错清单

- 需求分析阶段的成果通常是数据字典和数据流图，不是 E-R 图。
- 概念结构设计的成果是 E-R 图，独立于具体 DBMS。
- 逻辑结构设计才把 E-R 图转换为关系模型。
- 物理设计关注存储结构、存取方法、索引、聚簇，不是画 E-R 图。
- m:n 联系必须单独转换为关系模式，联系属性也放在该关系中。
- 1:n 联系通常把 1 端主键加入 n 端，不要反放。
- 弱实体关系模式的主键通常是强实体主键加弱实体部分键。
- 多值属性需要单独成表，不能直接塞进一个字段中。
- 自联系转换时要给两个角色的外键改名，避免混淆。
- 合并局部 E-R 图时要处理属性冲突、命名冲突和结构冲突。
- 建索引能加快查询，但会增加存储和更新维护代价。
- BCNF/3NF 属于逻辑模式优化内容，不是物理设计内容。
