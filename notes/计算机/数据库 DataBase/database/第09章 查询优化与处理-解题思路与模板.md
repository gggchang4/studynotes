# 第09章 查询优化与处理：开卷考试解题思路与模板

> 适用范围：查询编译、语法/语义分析、关系代数树、视图消解、基于规则的优化、基于代价的优化、基数估计、物理算子代价、连接算法、去重/分组/集合/投影/排序代价、计划枚举、查询执行、火山模型、即时编译。

## 0. 考场快速判题地图

| 题目关键词 | 题型 | 先套什么模板 |
| --- | --- | --- |
| SQL 如何变成执行计划 | 流程/概念题 | `SQL -> 解析树 -> 关系代数树 -> 逻辑计划 -> 物理计划 -> 执行` |
| 词法分析、语法分析、语义分析 | 编译流程题 | token、语法树、对象/权限/类型检查 |
| 视图、view expansion | 视图消解题 | 用视图定义替换视图名，再继续优化 |
| SQL 转关系代数树 | 转换题 | `SELECT -> σ/Π`，`FROM -> ×/⋈`，`GROUP BY -> γ`，`ORDER BY -> τ` |
| 逻辑优化、规则优化、RBO | 改写题 | 分解选择、推导谓词、下推选择、笛卡尔积转连接、下推投影 |
| 选择率、基数估计 | 计算题 | 先写统计信息，再选等值/范围/合取/析取/连接公式 |
| CBO、代价模型 | 比较/计算题 | `代价 = 子计划代价 + 当前物理算子代价` |
| 全表扫描、索引扫描、位图扫描 | 访问路径题 | 选择率低用索引，多条件低基数可用位图，无索引或返回多用全表扫 |
| NLJ、BNLJ、INLJ、MJ、HJ | 连接算法题 | 看内存 `M`、页数 `B(R)`、索引、是否有序、是否等值连接 |
| DISTINCT、GROUP BY、UNION | 物理实现题 | 排序法或哈希法；内存够约 `B(R)`，不够常见约 `3B(R)` |
| 外部排序、TPMMS | 排序代价题 | 初始归并段数 `ceil(B/M)`，每轮最多归并 `M-1` 个段 |
| 多表连接顺序 | 计划枚举题 | 左深树/右深树/稠密树；动态规划或贪心 |
| 火山模型、Iterator Model | 执行模型题 | 每个算子实现 `Open/Next/Close`，父算子向子算子拉取元组 |
| JIT、LLVM | 论述题 | 编译查询专用代码，减少解释执行、分支和函数调用开销 |

**本章总口诀：**

1. 逻辑层先“少读”：选择下推、投影下推、连接替代笛卡尔积。
2. 代价层先“估数”：基数估计决定后续 I/O、CPU 和中间结果大小。
3. 物理层先“选路”：访问路径、连接算法、排序/哈希实现。
4. 执行层先“流水”：火山模型拉取元组，JIT 用专用代码减少解释开销。

## 1. 查询处理总流程

### 1.1 标准流程

```text
SQL 查询
  -> Parser / Analyzer
  -> Parse Tree
  -> 关系代数表达式 / Logical Plan
  -> Logical Optimizer
  -> Physical Optimizer
  -> Physical Plan
  -> Executor
  -> 查询结果
```

答题时可以展开成：

1. **词法分析**：把 SQL 字符串切成 token，例如 `SELECT`、属性名、关系名、比较符、常量。
2. **语法分析**：根据 SQL 语法生成解析树，检查关键字和子句结构是否合法。
3. **语义分析**：检查表、列、类型、权限、完整性约束、作用域、别名等。
4. **逻辑计划生成**：把解析树转换成关系代数树。
5. **逻辑优化**：按等价规则改写关系代数树。
6. **物理优化**：为每个逻辑算子选择物理实现，并估算代价。
7. **查询执行**：按物理计划调用扫描、连接、排序、聚集等算子。

### 1.2 RBO 与 CBO 的区别

| 项目 | RBO：Rule Based Optimizer | CBO：Cost Based Optimizer |
| --- | --- | --- |
| 核心依据 | 关系代数等价规则和经验顺序 | 统计信息、基数估计、代价模型 |
| 典型动作 | 选择下推、投影下推、连接替换笛卡尔积 | 选择访问路径、连接算法、连接顺序 |
| 优点 | 简单、稳定、无需大量统计信息 | 更能适应数据分布和不同物理实现 |
| 风险 | 规则顺序可能影响质量 | 统计信息不准会导致差计划 |

答题句式：

```text
RBO 先保证逻辑等价并减少中间结果；CBO 再在等价物理计划中估算代价，选择总代价最低的计划。
```

## 2. 查询编译模板

### 2.1 SQL 到关系代数的对应表

| SQL 结构 | 关系代数算子 | 记忆方式 |
| --- | --- | --- |
| `SELECT A, B` | 投影 `Π_{A,B}` | 只保留输出列 |
| `FROM R, S` | 笛卡尔积 `R × S` | 未写连接条件时先看成乘积 |
| `WHERE 条件` | 选择 `σ_条件` | 过滤元组 |
| `JOIN ... ON` | 连接 `⋈_条件` | 选择 + 笛卡尔积的组合 |
| `NATURAL JOIN` | 自然连接 `⋈` | 同名属性自动等值连接 |
| `LEFT/RIGHT/FULL OUTER JOIN` | 外连接 | 保留一侧或两侧未匹配元组 |
| `UNION` | 并 `∪` | 默认去重 |
| `INTERSECT` | 交 `∩` | 共同元组 |
| `EXCEPT` | 差 `-` | 左边有右边无 |
| `DISTINCT` | 去重 `δ` | 删除重复元组 |
| `GROUP BY` | 分组/聚集 `γ` | 分组键 + 聚集函数 |
| `ORDER BY` | 排序 `τ` | 按指定属性排序 |

### 2.2 SQL 转关系代数树步骤

给定 SQL：

```sql
SELECT 输出属性
FROM R, S
WHERE R.a = S.a AND R.b > 10;
```

先写初始树：

```text
Π_输出属性
  σ_{R.a=S.a AND R.b>10}
    ×
   / \
  R   S
```

再优化为：

```text
Π_输出属性
  ⋈_{R.a=S.a}
 /          \
σ_{R.b>10}  S
    |
    R
```

考试答题模板：

```text
第一步：根据 FROM 构造叶子关系和初始笛卡尔积。
第二步：根据 WHERE 添加选择条件。
第三步：根据 SELECT 添加投影。
第四步：将连接条件从选择中拆出，把 σ(R×S) 改写为连接。
第五步：把只涉及单个关系的选择下推到对应关系上。
第六步：按需要下推投影，只保留输出属性和上层算子还要用到的属性。
```

### 2.3 视图消解模板

若有视图：

```sql
CREATE VIEW V AS
SELECT ...
FROM ...
WHERE ...
GROUP BY ...;
```

查询中出现：

```sql
SELECT ...
FROM R, V
WHERE ...
```

答题步骤：

```text
1. 把查询中的 V 替换为视图定义对应的关系代数表达式。
2. 将外层选择、投影、连接条件与视图内部表达式合并。
3. 再执行规则优化，例如选择下推、投影下推、连接顺序调整。
```

注意：

```text
视图消解不是执行视图，而是把视图定义展开为查询树的一部分。
```

## 3. 基于规则的优化 RBO

### 3.1 RBO 的目标

RBO 的核心不是计算精确代价，而是用一定顺序套用逻辑等价规则，使中间结果尽量小。

常见目标：

- 选择尽早做：减少元组数。
- 投影尽早做：减少属性宽度。
- 笛卡尔积尽量变成连接：避免巨大中间结果。
- 可合并的选择、投影、去重、排序尽量合并或删除。
- 子查询可改写为连接时，减少重复执行。

### 3.2 常用规则应用顺序

课件中的常用顺序可以记成：

```text
1. 分解复合的选择谓词
2. 添加可推导的谓词
3. 下推选择运算
4. 将笛卡尔积转换为连接
5. 引入并下推投影运算
```

答题时可以写：

```text
由于选择运算能减少元组数，所以优先下推；由于投影能减少元组宽度，所以在不丢失后续连接、选择、分组所需属性的前提下下推投影。
```

### 3.3 选择运算规则

复合选择分解：

```text
σ_{C1 AND C2}(R) = σ_{C1}(σ_{C2}(R))
```

选择下推到单个关系：

```text
若 C 只涉及 R 的属性：
σ_C(R ⋈ S) = σ_C(R) ⋈ S
σ_C(R × S) = σ_C(R) × S
```

选择和集合运算：

```text
σ_C(R ∪ S) = σ_C(R) ∪ σ_C(S)
σ_C(R - S) = σ_C(R) - σ_C(S)
```

笛卡尔积转连接：

```text
σ_{R.a = S.b}(R × S) = R ⋈_{R.a=S.b} S
```

谓词推导：

```text
若 A = B 且 A > 5，则可推出 B > 5。
若 R.a = S.a 且 R.a = 10，则可推出 S.a = 10。
```

### 3.4 投影运算规则

连续投影合并：

```text
Π_A(Π_B(R)) = Π_A(R)        其中 A ⊆ B
```

投影下推原则：

```text
下推到某个子树时，必须保留：
1. 最终输出需要的属性；
2. 上层选择条件要用的属性；
3. 上层连接条件要用的属性；
4. GROUP BY、ORDER BY、聚集函数要用的属性。
```

投影下推模板：

```text
原计划：
Π_L(R ⋈_{R.a=S.a} S)

可改写为：
Π_L(Π_{L_R ∪ {R.a}}(R) ⋈_{R.a=S.a} Π_{L_S ∪ {S.a}}(S))
```

其中 `L_R`、`L_S` 是最终输出中来自 `R`、`S` 的属性。

易错点：

```text
投影不能把连接属性、选择属性、分组属性提前删掉。
```

### 3.5 去重、分组和排序规则

去重：

```text
如果 R 本身无重复，则 δ(R) = R。
如果上层不要求集合语义，不能随意添加或删除 δ。
```

分组聚集：

```text
GROUP BY 下推或合并时，必须保证分组键和聚集函数含义不变。
```

排序：

```text
如果中间排序结果没有被 ORDER BY、Merge Join、GROUP BY 等后续算子利用，则通常可删除中间排序。
```

### 3.6 子查询改写模板

典型子查询：

```sql
SELECT Student.Sno, Sname, Sdept
FROM Student
WHERE Sno IN (
  SELECT Sno
  FROM SC
  WHERE Cno = '1'
);
```

可改写为连接：

```sql
SELECT Student.Sno, Sname, Sdept
FROM Student, SC
WHERE Student.Sno = SC.Sno
  AND Cno = '1';
```

答题句式：

```text
将相关或非相关子查询展开为连接后，可以让选择下推、连接重排和索引访问等优化统一参与计划选择。
```

### 3.7 常数传播和删除非必要谓词

常数传播：

```text
Student.Sno = SC.Sno AND Student.Sno < 100
=> 可推出 SC.Sno < 100
```

删除非必要谓词：

```text
MAX(DISTINCT Grade) = MAX(Grade)
MIN(DISTINCT Grade) = MIN(Grade)
```

原因：

```text
最大值、最小值不受重复值影响，DISTINCT 对结果没有贡献。
```

## 4. 基于代价的优化 CBO

### 4.1 CBO 总流程

```text
1. 收集统计信息
2. 对选择、投影、连接、分组等算子估计基数
3. 为逻辑算子选择物理实现
4. 枚举候选计划
5. 计算每个计划的总代价
6. 选择总代价最小的计划
```

答题句式：

```text
CBO 的关键是用统计信息估计中间结果大小，再结合 I/O、CPU、内存和通信代价选择物理计划。
```

### 4.2 统计信息符号表

| 符号 | 含义 |
| --- | --- |
| `T(R)` | 关系 `R` 的元组数 |
| `B(R)` | 存储关系 `R` 需要的页面数 |
| `L(R)` | `R` 中一个元组平均字节数 |
| `f(R)` | 一个页面能存储的 `R` 元组数 |
| `min(R,A)` | `R.A` 的最小值 |
| `max(R,A)` | `R.A` 的最大值 |
| `V(R,A)` | `R.A` 的不同取值个数；若 `A` 为属性集，表示 `Π_A(R)` 的不同元组数 |
| `MCV(R,A)` | 属性 `A` 上最常见的 Top-K 值 |
| `MCV_f(c)` | 最常见值 `c` 的出现频率 |
| `M` | 缓冲区页数 |

统计信息还可能包括：

- 直方图：描述属性值区间及其频率。
- 空值比例：`null_frac`。
- 多列统计信息：避免独立性假设带来的误差。

### 4.3 常用假设

| 假设 | 含义 | 风险 |
| --- | --- | --- |
| 均匀分布假设 | 属性值在取值范围内均匀出现 | 数据倾斜时误差大 |
| 属性独立假设 | 不同条件互不影响 | 多列强相关时误差大 |
| 值集包含假设 | 连接属性一侧值集包含另一侧 | 主键-外键连接较合理 |

答题补充句：

```text
如果统计信息过期、直方图过粗或属性之间存在相关性，基数估计会偏差，进而影响计划质量。
```

## 5. 选择运算的基数估计

### 5.1 基本概念

若：

```text
S = σ_C(R)
```

则：

```text
选择率 sel(C) = T(S) / T(R)
T(S) = T(R) * sel(C)
```

### 5.2 等值谓词

设：

```text
S = σ_{A=c}(R)
```

按统计信息优先级估计：

| 已知信息 | 估计公式 |
| --- | --- |
| `c` 是最常见值 | `T(S) = T(R) * MCV_f(c)` |
| 有直方图 | `T(S) = T(R) / (K * N)`，`K` 为桶数，`N` 为 `c` 所在桶的值个数 |
| 都没有 | `T(S) = T(R) / V(R,A)` |

### 5.3 不等值谓词

```text
S = σ_{A != c}(R)
T(S) = T(R) - T(σ_{A=c}(R))
```

若采用均匀分布：

```text
sel(A != c) = 1 - 1 / V(R,A)
T(S) = T(R) * (1 - 1 / V(R,A))
```

### 5.4 范围谓词

设：

```text
S = σ_{A<c}(R)
```

如果采用均匀分布：

```text
若 c < min(R,A)，T(S) = 0
若 c > max(R,A)，T(S) = T(R)
否则，T(S) = ((c - min(R,A)) / (max(R,A) - min(R,A))) * T(R)
```

同理：

```text
σ_{A>c}(R) 可用 ((max(R,A) - c) / (max(R,A) - min(R,A))) * T(R)
```

有直方图时：

```text
先统计完整落入范围的桶，再对边界桶按桶内均匀分布估算。
```

### 5.5 MCV、直方图和 NULL 修正

课件中的通用思路：

```text
selectivity = mcv_selectivity
            + hist_selectivity * (1 - sum(MCV_f))
```

考虑空值：

```text
selectivity = mcv_selectivity
            + hist_selectivity * (1 - sum(MCV_f) - null_frac)
```

使用场景：

- 条件命中 Top-K 常见值时，直接使用 MCV 频率。
- 条件落在非常见值区域时，用直方图估计剩余部分。
- 空值不满足普通比较谓词时，要扣除 `null_frac`。

### 5.6 多个选择条件

合取：

```text
S = σ_{F1 AND F2 AND ... AND Fn}(R)
sel(S) = f1 * f2 * ... * fn
T(S) = T(R) * f1 * f2 * ... * fn
```

析取：

```text
S = σ_{F1 OR F2 OR ... OR Fn}(R)
sel(S) = 1 - (1-f1)(1-f2)...(1-fn)
T(S) = T(R) * sel(S)
```

非：

```text
sel(NOT F) = 1 - sel(F)
```

易错点：

```text
合取/析取公式默认条件相互独立；如果题目给了多列统计信息，应优先使用多列统计信息。
```

## 6. 连接运算的基数估计

### 6.1 笛卡尔积

```text
T(R × S) = T(R) * T(S)
```

### 6.2 单属性自然连接 / 等值连接

设：

```text
U = R(X,Y) ⋈ S(Y,Z)
```

常用估计：

```text
T(R ⋈ S) = T(R) * T(S) / max(V(R,Y), V(S,Y))
```

推理方式：

```text
若 V(R,Y) >= V(S,Y)，则 S 中每个元组大约匹配 T(R)/V(R,Y) 个 R 元组。
若 V(R,Y) <= V(S,Y)，则 S 中每个元组大约匹配 T(R)/V(S,Y) 个 R 元组。
综合得到除以 max(V(R,Y), V(S,Y))。
```

### 6.3 多属性自然连接

设连接属性为 `y1, y2, ..., yn`，若假设连接条件相互独立：

```text
T(R ⋈ S) =
T(R) * T(S) / Π_i max(V(R, yi), V(S, yi))
```

### 6.4 主键-外键连接

若：

```text
S.fk -> R.pk
```

并且外键完整、非空，则：

```text
T(R ⋈ S) ≈ T(S)
```

答题解释：

```text
S 中每个外键元组至多匹配 R 中一个主键元组；若引用完整，每个 S 元组都能匹配。
```

### 6.5 连接估计注意事项

- 如果连接条件选择性强，实际连接基数可能更小。
- 如果两个关系用主键和外键连接，连接结果常接近外键表大小。
- 如果没有等值条件，应先看作笛卡尔积再加选择谓词。
- 数据倾斜会让哈希连接分区不均，也会影响基数估计。

## 7. 其他逻辑算子的基数估计

### 7.1 去重 `δ`

设 `R(a1, a2, ..., an)`：

```text
若 R 中无重复元组：
T(δ(R)) = T(R)

若 R 中元组全部相同：
T(δ(R)) = 1
```

粗略估计：

```text
T(δ(R)) = min(T(R)/2, Π_i V(R, ai))
```

### 7.2 分组聚集 `γ`

若分组属性为 `A`：

```text
T(γ_A(R)) = V(R,A)
```

若无法获得完整多列不同值个数：

```text
T(γ_{a1,...,an}(R)) = min(T(R)/2, Π_i V(R, ai))
```

### 7.3 集合运算

若两个输入是不同关系 `R`、`S`：

并：

```text
UNION ALL: T = T(R) + T(S)
UNION: T 的范围约为 [max(T(R),T(S)), T(R)+T(S)]
课件粗估：T ≈ (max(T(R),T(S)) + T(R) + T(S)) / 2
```

交：

```text
INTERSECT: T 的范围约为 [0, min(T(R),T(S))]
课件粗估：T ≈ min(T(R),T(S)) / 2
```

差：

```text
EXCEPT / R - S: T 的范围约为 [max(0, T(R)-T(S)), T(R)]
课件粗估时可先写范围，再按题目要求取中间估计值。
```

## 8. 访问路径与物理算子选择

### 8.1 全表扫描 Full Table Scan

含义：

```text
顺序读取关系 R 的所有数据页，检查每个元组是否满足 WHERE 条件。
```

代价：

```text
Cost ≈ B(R)
```

适用：

- 没有可用索引。
- 查询会访问表中大多数页面。
- 表很小。

### 8.2 索引扫描 Index Scan

含义：

```text
先通过索引定位满足条件的记录位置，再根据位置回表读取数据页。
```

代价组成：

```text
Cost ≈ 索引访问页数 + 回表读取页数
```

特殊情况：

```text
覆盖索引 Cover Index：查询所需属性都在索引中，不需要回表。
```

适用：

- 选择率低。
- 返回元组较少。
- 条件属性上有索引。

### 8.3 位图扫描 Bitmap Scan

含义：

```text
先用多个索引得到满足各条件的元组位置位图，再对位图做 AND/OR，最后按页面顺序回表。
```

代价：

```text
Cost ≈ 扫描索引页数 + 回表页数
```

适用：

- 多个条件组合。
- 属性基数较低。
- 需要减少随机回表，把回表整理成较顺序的页面访问。

### 8.4 访问路径选择口诀

```text
返回少：优先索引。
返回多：优先全表扫描。
多条件低基数：考虑位图扫描。
索引覆盖：索引扫描优势明显。
```

## 9. 连接算法与代价模板

设：

```text
B(R), B(S)：关系页数
T(R), T(S)：元组数
M：可用缓冲区页数
```

### 9.1 嵌套循环连接 NLJ

基本思想：

```text
for each tuple r in R:
  for each tuple s in S:
    if join_condition(r,s):
      output(r,s)
```

若 `R` 为外表、`S` 为内表，最差 I/O：

```text
Cost ≈ B(R) + T(R) * B(S)
```

如果一个关系可全部放入内存：

```text
Cost ≈ B(R) + B(S)
```

适用：

- 外表很小。
- 没有可用索引和排序。
- 作为最基础、最通用的连接算法。

### 9.2 块嵌套循环连接 BNLJ

思想：

```text
一次读入外表的 M-2 个页面作为块，再扫描内表进行连接。
```

若 `O` 为外表、`I` 为内表：

```text
Cost ≈ B(O) + ceil(B(O)/(M-2)) * B(I)
```

课件中若选择较小的 `S` 作外表：

```text
Cost ≈ B(S) + ceil(B(S)/(M-2)) * B(R)
```

若外表可放入内存：

```text
Cost ≈ B(R) + B(S)
```

答题策略：

```text
通常选择页数更少的关系作外表，以减少扫描内表的次数。
```

### 9.3 索引嵌套循环连接 INLJ

适用条件：

```text
内表连接属性上有索引。
```

代价模板：

```text
Cost ≈ B(外表)
      + T(外表) * (每次索引查找代价 + 读取匹配数据页代价)
```

适用：

- 外表较小。
- 内表连接属性有索引。
- 每个外表元组能匹配的内表元组较少。

答题句式：

```text
INLJ 避免了对内表的反复全表扫描，把每个外表元组的匹配查找转换为索引查找。
```

### 9.4 归并连接 Merge Join

适用：

```text
等值连接或自然连接，并且两个输入在连接属性上有序，或排序代价可接受。
```

若两个输入已经有序：

```text
Cost ≈ B(R) + B(S)
```

若未排序：

```text
Cost ≈ Sort(R) + Sort(S) + B(R) + B(S)
```

注意：

```text
如果连接属性存在重复值，可能需要回退或缓存一侧相同连接键的元组。
```

### 9.5 哈希连接 Hash Join

适用：

```text
等值连接或自然连接。
```

基本思想：

```text
Build 阶段：对较小关系按连接属性建立哈希表。
Probe 阶段：扫描另一关系，用相同哈希函数查找匹配元组。
```

若较小关系能放入内存：

```text
Cost ≈ B(R) + B(S)
```

### 9.6 Grace Hash Join

阶段：

```text
阶段一：用哈希函数把 R 和 S 分别划分成 K 个分区。
阶段二：对每对对应分区 ri、si 做连接，再合并结果。
```

代价：

```text
分区阶段：读写 R、S，一次读入和一次写回，2 * (B(R)+B(S))
连接阶段：再次读入各分区，B(R)+B(S)
总 I/O：3 * (B(R)+B(S))
```

注意：

```text
如果分区仍然放不进内存，需要递归分区。
如果连接键分布严重倾斜，可能出现分区溢出。
```

### 9.7 Hybrid Hash Join

思想：

```text
在分区时保留一部分分区在内存中，另一关系对应分区到来时可直接连接输出，减少读写。
```

适用：

```text
内存不够容纳整个构建关系，但足够保留部分分区。
```

### 9.8 连接算法选择表

| 条件 | 优先考虑 |
| --- | --- |
| 外表很小、内表有索引 | INLJ |
| 等值连接、较小关系能进内存 | Hash Join |
| 等值连接、两个输入已按连接键有序 | Merge Join |
| 内存能容纳外表块，且无索引 | BNLJ |
| 没有索引、无序、通用兜底 | NLJ / BNLJ |
| 数据严重倾斜 | 谨慎使用 Hash Join，注意溢出处理 |

## 10. 去重、分组、集合、投影、排序的物理实现

### 10.1 去重 DISTINCT

排序法：

```text
按所有属性排序，相同元组相邻，扫描时删除重复元组。
```

哈希法：

```text
按哈希函数分区，每个分区内建立哈希表去重。
```

代价粗估：

```text
若 R 可放入内存：Cost ≈ B(R)
若不能放入内存：Cost ≈ 3 * B(R)
```

### 10.2 分组聚集 GROUP BY

排序法：

```text
按分组属性排序，同组元组相邻，顺序扫描并维护聚集状态。
```

哈希法：

```text
按分组属性哈希分区，对每个组维护聚集状态。
```

聚集函数常见状态：

```text
INITCOND：初始状态
SFUNC：每读入一个元组更新状态
FINALFUNC：输出最终聚集值
```

代价粗估：

```text
若可放入内存：Cost ≈ B(R)
若不能放入内存：Cost ≈ 3 * B(R)
```

### 10.3 集合运算

排序法：

```text
先对 R、S 排序，再归并扫描。
UNION：相同元组只保留一个。
INTERSECT：只输出两边都出现的元组。
EXCEPT：只输出 R 中出现但 S 中不出现的元组。
```

哈希法：

```text
用相同哈希函数划分 R、S。
UNION：把两边分区合并并去重。
INTERSECT：用一边建哈希表，另一边探测命中者。
EXCEPT：用 S 建哈希表，从 R 中删除命中者。
```

### 10.4 投影代价估算

投影不仅会减少列，还会改变每页可容纳元组数。

模板：

```text
1. 计算投影后每个元组长度 L(Π(R))。
2. 计算每页可容纳元组数 f(Π(R))。
3. 计算投影结果页数 B(Π(R)) = ceil(T(Π(R)) / f(Π(R)))。
```

若投影包含去重：

```text
还要叠加 DISTINCT 的排序或哈希去重代价。
```

### 10.5 排序算子

内存排序：

```text
若待排序数据能全部放入内存，直接内存排序。
```

外部排序 TPMMS：

```text
阶段一：每次读入 M 页，在内存中排序，写回一个有序归并段。
阶段二：每次最多归并 M-1 个有序段，反复归并直到只剩一个有序结果。
```

常用计算：

```text
初始归并段数 = ceil(B / M)
每轮最多归并段数 = M - 1
归并轮数 = ceil(log_{M-1}(ceil(B/M)))
若每轮都读写，总代价约为 2B * (1 + 归并轮数)
若最后一轮直接流水输出，可少写一次最终结果。
```

考试提示：

```text
题目给 B 和 M 时，先算初始段数，再算需要几轮把这些段归并成 1 段。
```

## 11. 计划代价估算模板

### 11.1 自底向上估计

对一棵物理计划树：

```text
1. 从叶子节点开始估计 T、B 和访问代价。
2. 对每个内部节点估计输出基数。
3. 根据所选物理算子计算当前节点代价。
4. 累加子计划代价和当前算子代价。
5. 根节点得到最终计划代价。
```

可写成：

```text
Cost(node) =
  Cost(left child)
+ Cost(right child)
+ Cost(current physical operator)
```

### 11.2 比较两个计划的答题模板

```text
计划 A：
1. 先执行 ...
2. 中间结果基数为 ...
3. 使用 ... 算法，代价为 ...
4. 总代价为 ...

计划 B：
1. 先执行 ...
2. 中间结果基数为 ...
3. 使用 ... 算法，代价为 ...
4. 总代价为 ...

比较：
由于 ... 的中间结果更小 / 扫描次数更少 / 可利用索引 / 避免排序，
所以选择计划 ...
```

### 11.3 统计信息精度对计划质量的影响

统计信息维护内容：

- 插入或删除后维护 `T(R)`。
- 数据页变化后维护 `B(R)`。
- 属性取值变化后维护 `V(R,A)`。
- 定期更新直方图、MCV 和 NULL 比例。

影响因素：

- 被选择的逻辑查询计划。
- 中间关系的大小。
- 物理操作符的实现代价。
- 相邻操作符的关系，例如排序结果可被 Merge Join 复用。
- 一个操作符传递给下一个操作符的参数。

答题句式：

```text
统计信息不准确会造成基数估计错误，优化器可能低估中间结果，进而选择错误的连接顺序或连接算法。
```

## 12. CBO 中计划枚举与生成

### 12.1 多表连接树

| 树形 | 特点 |
| --- | --- |
| 左深树 | 每个内部节点的右孩子是基关系；常见、搜索空间较小、便于流水执行 |
| 右深树 | 每个内部节点的左孩子是基关系 |
| 稠密树 / bushy tree | 左右孩子都可能是中间结果；搜索空间最大，可能找到更优计划 |

### 12.2 动态规划算法

核心思想：

```text
对子集合 S 保存最优连接代价 C(S) 和最优连接顺序 Seq(S)。
由小集合逐步推出大集合。
```

状态转移：

```text
若 |S| = 1：
  C(S) = 单表访问代价

若 |S| > 1：
  枚举 S 的非空真子集 S_l
  S_r = S - S_l
  C(S) = min( C(S_l) + C(S_r) + Cost(S_l, S_r) )
  Seq(S) = Seq(S_l) ⋈ Seq(S_r)
```

左深树限制下：

```text
每次只枚举一个基表 A 作为右侧输入：
C(S) = min( C(S-A) + C(A) + Cost(S-A, A) )
```

答题步骤：

```text
1. 列出所有单关系的 T、V、访问代价。
2. 枚举所有大小为 2 的关系集合，保存最优连接顺序。
3. 枚举大小为 3、4、... 的集合。
4. 每一步只保留该集合的最小代价计划。
5. 到全集时得到最优连接顺序。
```

### 12.3 贪心算法

思想：

```text
每次选择当前状态下增量代价最小的关系连接。
```

步骤：

```text
1. 选择一个大小最小或代价最低的关系作为当前关系 R_cur。
2. 从未连接关系中选择与 R_cur 连接后结果最小的 R_next。
3. 更新 R_cur = R_cur ⋈ R_next。
4. 重复直到所有关系都被连接。
```

优点：

```text
搜索空间小，速度快。
```

缺点：

```text
只能得到局部最优，不能保证全局最优。
```

### 12.4 遗传算法

概念题答法：

```text
遗传算法是一种启发式优化方法。一个个体表示一种连接顺序，适应度为该计划的代价。
通过交叉、变异和选择产生新一代计划，保留代价较低的计划。
```

优缺点：

```text
优点：不受关系数量限制太严重，适合搜索空间很大的场景。
缺点：随机性强，结果不确定，不能保证得到最优计划。
```

## 13. 查询执行与火山模型

### 13.1 火山模型

火山模型又称：

```text
Iterator Model / Pipeline Model
```

基本接口：

```text
Open()：初始化算子
Next() / GetNext()：返回下一个结果元组
Close()：释放资源
```

执行方式：

```text
父算子调用子算子的 Next()。
子算子按需返回一个元组。
数据从叶子扫描算子向根算子逐步流动。
```

答题句式：

```text
火山模型以算子为核心，每个算子实现统一接口，上层算子通过 Next() 拉取下层结果，因此天然支持流水线执行。
```

### 13.2 示例执行过程

查询：

```sql
SELECT Sno, Sname, Grade
FROM SC, Student
WHERE SC.Sno = Student.Sno
  AND Cno = '1';
```

可能的执行树：

```text
Π_{Sno,Sname,Grade}
  ⋈_{SC.Sno=Student.Sno}
 /                         \
Student                 σ_{Cno='1'}(SC)
```

火山模型过程：

```text
1. 根投影算子调用连接算子的 Next()。
2. 连接算子调用左右孩子的 Next()。
3. SC 分支先过滤 Cno='1'。
4. 连接算子找到匹配元组后返回给投影算子。
5. 投影算子裁剪属性并向上返回一个结果元组。
6. 重复 Next()，直到没有更多结果。
```

### 13.3 流水线与物化

流水线执行：

```text
中间结果不完全落盘，元组一边产生一边被上层消费。
```

物化执行：

```text
先把某个中间结果完整生成并存储，再交给上层算子使用。
```

比较：

| 方式 | 优点 | 缺点 |
| --- | --- | --- |
| 流水线 | 减少中间结果存储，响应更快 | 某些阻塞算子不适合 |
| 物化 | 简单，结果可重复使用 | 可能产生大量 I/O |

阻塞算子例子：

```text
排序、去重、某些聚集通常需要看到较多输入后才能输出。
```

## 14. 即时编译 JIT

### 14.1 基本概念

JIT：

```text
Just-In-Time Compiler，即时编译器。
```

在查询执行中的作用：

```text
根据当前查询的表达式、操作符和数据布局生成专用机器码，用编译执行替代解释执行。
```

课件例子中的表达式可理解为：

```text
row.getFloat("totalprice") > 5000
AND
row.getString("c_clerk") == "zhangsan"
```

JIT 可以把这种表达式编译成更直接的判断代码。

### 14.2 为什么能加速

常见收益：

- 减少解释器循环开销。
- 减少虚函数调用和分支判断。
- 根据已知类型和表达式生成专用代码。
- 更容易被 CPU 缓存和流水线优化。
- 可借助 LLVM 等编译框架生成高效机器码。

答题句式：

```text
JIT 在编译期利用查询计划中已有信息生成专用代码，减少执行期反复判断和解释，从而提升 CPU 利用率。
```

### 14.3 JIT 适用场景

适合：

- 表达式计算密集。
- 过滤条件复杂。
- 查询耗时主要在 CPU 而不是 I/O。
- 重复执行或处理大量元组的查询。

不一定明显：

- 查询主要受磁盘 I/O 限制。
- 查询很短，编译开销超过收益。

## 15. 高频题型模板

### 15.1 SQL 转逻辑计划题

```text
题目：给 SQL，画出初始查询树并优化。

答题：
1. FROM 中每个表作为叶子节点。
2. 多表 FROM 先写成 ×。
3. WHERE 写成 σ。
4. SELECT 写成 Π。
5. GROUP BY 写成 γ，ORDER BY 写成 τ，DISTINCT 写成 δ。
6. 拆分 WHERE 中的 AND 条件。
7. 单表条件下推到对应叶子。
8. 连接条件把 × 改成 ⋈。
9. 投影下推，保留输出属性、连接属性和过滤属性。
```

### 15.2 RBO 改写题

```text
题目：用规则优化关系代数表达式。

答题：
1. 分解复合选择：σ_{C1 AND C2} -> σ_C1 σ_C2。
2. 推导新谓词：A=B 且 A=c -> B=c。
3. 下推选择：只涉及单关系的条件放到该关系上。
4. 连接替换：σ_join(R×S) -> R ⋈ S。
5. 下推投影：只保留输出和后续算子需要的属性。
6. 删除冗余算子：无意义 DISTINCT、重复投影、中间排序等。
7. 说明改写前后逻辑等价，但中间结果更小。
```

### 15.3 基数估计题

```text
题目：给 T(R)、V(R,A)、min/max、直方图等，估计选择或连接结果大小。

答题：
1. 写出原关系基数 T(R)。
2. 判断谓词类型：等值 / 不等值 / 范围 / 合取 / 析取 / 连接。
3. 选择公式：
   等值：T/V 或 T*MCV_f(c)
   范围：按 min/max 比例或直方图
   合取：选择率相乘
   析取：1 - Π(1-fi)
   连接：T(R)T(S)/max(V(R,A),V(S,A))
4. 算出选择率和输出基数。
5. 若多步计划，则把上一步输出作为下一步输入。
```

### 15.4 连接算法代价题

```text
题目：比较 NLJ、BNLJ、INLJ、MJ、HJ 的代价。

答题：
1. 写出 B(R)、B(S)、T(R)、T(S)、M。
2. 判断是否有索引、是否有序、是否等值连接。
3. 代入公式：
   NLJ: B(O) + T(O)*B(I)
   BNLJ: B(O) + ceil(B(O)/(M-2))*B(I)
   INLJ: B(O) + T(O)*(索引查找+回表)
   MJ: Sort(O)+Sort(I)+B(O)+B(I)
   HJ: 内存够 B(O)+B(I)，Grace Hash 约 3*(B(O)+B(I))
4. 选择代价最小者，并说明原因。
```

### 15.5 外部排序题

```text
题目：B 页数据，M 页缓冲，计算 TPMMS 轮数和 I/O。

答题：
1. 初始归并段数 r = ceil(B/M)。
2. 每轮最多归并 M-1 个段。
3. 归并轮数 p = ceil(log_{M-1}(r))。
4. 若每轮读写，总代价约 2B*(1+p)。
5. 若最后结果直接输出给上层，可说明最后一轮可能少写一次。
```

### 15.6 多表连接顺序题

```text
题目：给多个关系和统计信息，求较优连接顺序。

动态规划答题：
1. 先列单关系的 T、V、C、Seq。
2. 枚举两两连接，计算基数和代价，保留每个集合的最优。
3. 枚举三表、四表连接，继续只保留每个集合的最优。
4. 到全集时输出 Seq(全集)。

贪心答题：
1. 选择当前最小关系或最小中间结果作为起点。
2. 每次加入使中间结果增加最少的关系。
3. 重复直到全部连接。
4. 说明贪心速度快但不保证全局最优。
```

### 15.7 执行模型/JIT 论述题

火山模型：

```text
数据库把查询计划看作算子树，每个算子提供 Open、Next、Close 接口。
根算子不断调用子算子的 Next，子算子再递归调用自己的孩子。
这种拉取式执行便于把不同物理算子组合起来，并支持流水线执行。
```

JIT：

```text
即时编译在查询执行前或执行中，根据查询表达式和数据类型生成专用机器码。
它减少解释执行中的重复判断、函数调用和分支开销，适合 CPU 密集型表达式计算。
实际系统可借助 LLVM 等框架生成高效代码。
```

## 16. 易错清单

- `FROM R,S WHERE R.a=S.a` 初始可写 `σ(R×S)`，优化后应写成连接。
- 选择下推只适用于条件涉及的属性都在该子树中。
- 投影下推不能删掉连接属性、选择属性、分组属性、排序属性。
- `MAX(DISTINCT A)` 和 `MIN(DISTINCT A)` 中的 `DISTINCT` 通常不影响结果。
- 等值选择没有 MCV、直方图时才用 `T(R)/V(R,A)`。
- 合取选择率相乘依赖独立性假设，属性相关时误差可能很大。
- 自然连接基数常用 `T(R)T(S)/max(V(R,A),V(S,A))`，不是除以两个不同值个数的乘积，除非是多连接属性。
- BNLJ 的外表应尽量选页数小的关系。
- INLJ 是否划算取决于外表元组数和内表索引查找代价。
- Merge Join 便宜的前提是输入已排序，未排序要加排序代价。
- Hash Join 主要适用于等值连接，不适合任意不等值连接。
- Grace Hash Join 常见总 I/O 是 `3*(B(R)+B(S))`，但数据倾斜或递归分区会增加代价。
- `UNION ALL` 不去重，`UNION` 去重。
- 外部排序要区分初始排序段生成和后续归并轮数。
- 动态规划只保留每个关系集合的最优计划，避免重复计算子问题。
- 火山模型是拉取式 `Next()`，不是一次性把所有结果都先生成。
- JIT 优化 CPU 开销，不一定能显著改善 I/O 瓶颈查询。

## 17. 一页速记版

```text
查询处理：
SQL -> 解析/语义分析 -> 关系代数树 -> RBO -> CBO -> 物理计划 -> 执行

RBO：
分解选择 -> 推导谓词 -> 选择下推 -> × 转 ⋈ -> 投影下推

选择基数：
等值：T/V 或 T*MCV_f(c)
范围：(c-min)/(max-min)*T
合取：T * f1*f2*...
析取：T * (1-(1-f1)(1-f2)...)

连接基数：
T(R⋈S)=T(R)T(S)/max(V(R,A),V(S,A))
多属性连接除以每个连接属性 max(V) 的乘积
主键-外键连接结果常接近外键表大小

访问路径：
全表扫 B(R)
索引扫 = 索引页 + 回表页
位图扫 = 多索引位图合并 + 回表页

连接代价：
NLJ: B(O)+T(O)B(I)
BNLJ: B(O)+ceil(B(O)/(M-2))B(I)
INLJ: B(O)+T(O)*(索引查找+回表)
MJ: sort(R)+sort(S)+B(R)+B(S)
HJ: 内存够 B(R)+B(S)，GHJ 约 3(B(R)+B(S))

排序：
r=ceil(B/M)
fan-in=M-1
p=ceil(log_{M-1}(r))
Cost≈2B(1+p)

执行：
火山模型 Open/Next/Close，父算子拉取子算子结果。
JIT 用查询专用机器码减少解释执行和分支/调用开销。
```
