# Agent项目架构

## 本笔记的学习目标
- 理解生产级 Agent 项目的模块划分、代码组织和工程边界。

## 核心内容
- Agent 项目不应只有一个 Prompt 文件和一段模型调用代码，而应按职责拆分为模型层、Prompt 层、工具层、状态层、记忆层、评估层、观测层和业务接口层。
- Datawhale 自建 Agent 框架章节强调 LLM 封装、Agent 范式、工具系统和执行器；Microsoft 生产化课程强调托管运行、状态、观测和安全。
- 好的项目架构能让 Prompt、工具、模型、评估和业务逻辑各自演进，而不是互相纠缠。

## 推荐模块
- `llm/`：模型客户端、模型路由、重试和降级。
- `prompts/`：System Prompt、任务模板、版本管理。
- `tools/`：工具定义、Schema、权限、执行器。
- `agents/`：Agent 核心循环、不同 Agent 类型。
- `workflows/`：确定性流程、图编排、人工节点。
- `memory/`：短期记忆、长期记忆、用户画像。
- `rag/`：文档入库、检索、重排、引用。
- `evals/`：测试集、评估器、回归 Harness。
- `observability/`：日志、Trace、指标、成本。
- `api/`：服务接口、鉴权、任务队列。

## 工程化要点
- Prompt 和工具 Schema 要能被单独测试。
- Agent 执行状态要可持久化，便于恢复和排查。
- 模型调用、工具调用、RAG 检索必须进入统一 Trace。
- 业务动作要和模型推理隔离，模型只能提出动作请求，程序负责校验和执行。
- 高风险工具需要权限策略和人工确认。
