# Agent安全与治理

## 本笔记的学习目标
- 理解 Agent 系统在安全、权限、隐私、审计和人工复核上的工程要求。

## 核心内容
- Agent 安全风险来自模型不确定性、工具执行能力、外部内容注入、长期记忆、跨系统协议和自动化行动。
- Microsoft 课程有 Trustworthy Agents 和 Securing Agents，说明安全不是上线后补丁，而是 Agent 设计的一部分。
- 安全治理的目标不是让 Agent 永远不出错，而是限制出错范围、可追踪、可恢复、可人工接管。

## 主要风险
- Prompt Injection：用户或外部资料诱导模型忽略系统指令。
- Tool 越权：模型调用了不该调用的工具或传入危险参数。
- 数据泄露：把敏感信息发给模型、工具或无权限用户。
- 记忆污染：把错误或恶意信息写入长期记忆。
- 自动化误操作：发送邮件、删除数据、修改权限等动作不可逆。
- 协议暴露面：MCP、A2A、浏览器 Agent 暴露资源或工具过宽。

## 防护措施
- 指令分级：系统规则高于用户输入和外部内容。
- 最小权限：工具、资源、用户、会话分别授权。
- 参数校验：危险参数和范围由程序拦截。
- 人工确认：高风险操作必须暂停确认。
- 数据脱敏：敏感数据进入模型前处理。
- 审计日志：记录谁、何时、为什么、调用了什么工具、产生了什么结果。
- 红队测试：用恶意输入和越权场景测试系统。

## 工程化理解
- 不能把安全完全交给 Prompt。
- 外部文档和网页内容只能作为数据，不能成为指令。
- Web Agent 和工具调用 Agent 必须有更严格的权限和审计。
- 合规场景要保留引用、判断依据和人工复核记录。
