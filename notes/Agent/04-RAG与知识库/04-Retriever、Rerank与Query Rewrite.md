# Retriever、Rerank与Query Rewrite

## 本笔记的学习目标
- 理解检索器、重排和查询改写如何共同提升 RAG 的召回率与答案质量。

## 核心内容
- Retriever 负责从知识库中召回候选内容，是 RAG 的第一道质量门。
- Rerank 负责对候选内容重新排序，筛掉语义相似但不真正有用的片段。
- Query Rewrite 负责把用户问题改写成更适合检索的查询，尤其适合口语化、多轮上下文和复杂问题。
- Agentic RAG 中，Agent 可以决定是否改写查询、是否多轮检索、是否换检索策略。

## Retriever 类型
- 向量检索：适合语义相似召回。
- 关键词检索：适合术语、编号、精确短语。
- 混合检索：结合语义和关键词，生产场景常用。
- 多路检索：从多个知识源、多个索引或多个 Query 同时召回。
- 结构化检索：通过 SQL、过滤条件或业务 API 查询。

## Query Rewrite 场景
- 多轮对话中的省略问题，例如“那它的限制是什么”。
- 用户问题过长，需要拆分为多个检索子问题。
- 用户使用俗称，需要改写为文档中的正式术语。
- 需要把答案任务和检索任务分离，例如“帮我写方案”先改写成资料检索问题。

## 工程化理解
- Top-K 不是越大越好，召回过多会污染上下文。
- Rerank 可以显著提升质量，但会增加延迟和成本。
- Query Rewrite 本身也会出错，需要记录原问题、改写问题和检索结果。
- 检索质量要单独评估，不能只看最终答案。
