# Embedding与向量数据库

## 本笔记的学习目标
- 理解 Embedding、向量检索和向量数据库在 RAG 中的作用与选型要点。

## 核心内容
- Embedding 是把文本、图片或其他对象编码成向量，使系统可以用相似度进行检索。
- 向量数据库负责存储向量、元数据和索引，并支持相似度检索、过滤、更新和删除。
- 向量检索适合语义相似问题，但不一定适合精确关键词、数字、编号和权限过滤。
- 生产 RAG 通常需要混合检索，把向量召回、关键词检索和元数据过滤结合起来。

## 关键概念
- 向量维度：不同 Embedding 模型输出维度不同。
- 相似度：常见有 cosine、dot product、L2。
- 索引：HNSW、IVF 等索引用于加速近似最近邻搜索。
- 元数据过滤：按用户权限、文档类型、时间、业务标签过滤。
- Top-K：召回数量，需要与 Rerank 和上下文预算配合。

## 选型要点
- Embedding 模型是否支持中文、代码、领域术语和长文本。
- 向量数据库是否支持权限过滤、增量更新、备份和多租户。
- 是否需要本地部署、云服务、混合检索或稀疏向量。
- 检索延迟和索引构建成本是否符合业务要求。

## 工程化理解
- Embedding 模型升级会导致向量空间变化，通常需要重建索引。
- 向量召回不保证事实正确，最终答案仍需引用和评估。
- 对精确条款、金额、日期等问题，应结合关键词和结构化查询。
