# 上下文管理与结构化输出

## 本笔记的学习目标
- 理解如何管理模型上下文，并让模型输出稳定、可解析、可进入业务流程的结果。

## 核心内容
- 上下文管理是把正确的信息在正确时刻放进模型输入，而不是把所有历史、文档和规则无差别塞进去。
- 上下文通常包含系统指令、用户输入、历史消息、任务状态、检索结果、工具结果、记忆和输出格式约束。
- 结构化输出是把模型回答变成业务系统可消费的数据，例如 JSON、枚举、评分、计划步骤、工具参数或审批结果。
- Datawhale 的上下文工程强调信息分层、压缩、回查、结构化笔记和工具辅助探索；Microsoft 课程也把 Context Engineering 作为独立主题。

## 上下文策略
- 分层：系统规则、任务输入、历史信息、检索资料、工具结果分层管理。
- 裁剪：根据任务相关性保留信息，而不是按时间机械截断。
- 摘要：长期历史用摘要保留关键事实、决策和未完成事项。
- 回查：需要精确信息时通过工具或 RAG 重新获取，而不是依赖压缩后的记忆。
- 优先级：安全规则、用户明确约束和当前任务目标优先级最高。

## 结构化输出策略
- 用 Schema 明确字段名、类型、枚举、必填项和约束。
- 对关键字段做程序校验，不通过则触发修复或人工确认。
- 把工具调用参数与最终回答分开，避免模型把解释文本混入参数。
- 对高风险操作要求输出理由、置信度和待确认事项。

## 工程化理解
- Context Engineering 的本质是注意力调度，不只是扩大上下文窗口。
- 输出结构越明确，后续评估、日志和业务集成越容易。
- 长上下文模型不能替代检索、摘要、状态管理和权限控制。
