# LLM应用基本架构

## 本笔记的学习目标
- 理解一个 LLM 应用从用户输入到模型调用、上下文构造、输出处理和业务落地的基本链路。

## 核心内容
- LLM 应用通常包含输入层、上下文构造层、模型调用层、输出解析层、业务执行层和观测评估层。
- 输入层负责接收用户请求、文件、业务事件或系统消息，并进行基础校验。
- 上下文构造层把系统指令、用户问题、历史消息、RAG 结果、工具说明、业务约束组装成模型可用的 Prompt。
- 模型调用层负责选择模型、传入参数、处理流式输出、超时、重试和降级。
- 输出解析层把自然语言、JSON、Tool Call 或其他结构化结果转为业务可用的数据。
- 业务执行层负责调用工具、写数据库、触发工作流或返回用户。
- 观测评估层负责记录 Prompt、响应、Token、延迟、错误、Trace 和评估结果。

## 架构模式
- 简单问答：输入 -> Prompt -> LLM -> 输出。
- RAG 问答：输入 -> Query Rewrite -> 检索 -> 重排 -> Prompt -> LLM -> 带引用输出。
- 工具调用：输入 -> 工具 Schema -> LLM 选择工具 -> 执行工具 -> LLM 汇总。
- Agent：输入 -> 状态初始化 -> Think/Act/Observe 循环 -> 停止条件 -> 输出。

## 工程化理解
- Prompt 不应该散落在代码中，需要版本管理和评估。
- 模型返回结果不能直接相信，需要结构化校验和业务规则兜底。
- LLM 应用上线前至少要有日志、评估集、错误处理、成本监控和安全边界。
